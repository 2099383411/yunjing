# 密码学攻击面推导 — 从密码学原语到攻击向量

> 关联 Level 0: `crypto/05-cryptography-security.md`
> 关联 Level 2: Case 9 (ZeroLogon), Case 10 (Heartbleed)
>
> 目标：枚举每个密码学原语的信任假设，标注假设被突破后的攻击向量。

---

## 一、对称加密的信任假设

### 信任假设 A：密钥是保密的

- **假设内容**：加密和解密所需的对称密钥只有授权方知道。
- **机制来源**：对称加密的安全性完全依赖密钥的保密性。
- **如果被突破**：密钥泄露 → 所有用该密钥加密的数据可解密。

| 突破方式 | 推导的攻击 | 实际案例 |
|---------|-----------|---------|
| 密钥硬编码在代码中 | 反编译 → 提取密钥 | 移动应用/嵌入式设备 |
| 密钥存储在可访问的文件 | 文件读取 → 密钥泄露 | 配置文件中明文密钥 |
| 密钥通过不安全信道传输 | 网络嗅探 → 密钥泄露 | 未加密的管理接口 |
| 密钥派生自弱密码 | 暴力破解 → 推导密钥 | 弱密码加密 |
| 密钥未按计划轮换 | 长期使用的密钥被收集足够多的密文 → 分析攻击 | 长期密钥不换 |

**推理模式：** 对称加密把"安全"的负担集中到了"密钥管理"上。算法本身可以被公开分析——安全性不在算法中，在密钥中。密钥管理 = 密码系统中最脆弱的部分。

### 信任假设 B：加密模式使用正确

- **假设内容**：开发人员选择了正确的加密模式（GCM/CTR 而不是 ECB/CBC 不带 MAC）。
- **机制来源**：分组密码有多种工作模式，各有不同的安全属性。
- **如果被突破**：密文可被篡改或部分解密。

| 突破方式 | 推导的攻击 | 实际案例 |
|---------|-----------|---------|
| ECB 模式 → 相同明文块产生相同密文块 | 明文模式识别 → 图片轮廓可见 | ECB 企鹅图 |
| CBC 模式无独立 IV → IV 固定或可预测 | 相同前缀的密文相同 → 数据泄露 | Cookie 加密 |
| CBC 模式无 MAC → 密文可被篡改 | 比特翻转 → 解密后的明文被修改 | CBC 比特翻转攻击 |
| 流密码重用 nonce | 两个密文 XOR → 明文 XOR → 可恢复 | OTP 重用 |

**推理模式：** 加密模式的选择决定了"在什么条件下加密是安全的"。ECB 不安全是因为它泄漏了明文的结构信息。CBC 需要 MAC 是因为它不防篡改。GCM/ChaCha20-Poly1305 提供 AEAD（认证加密）—— 加密 + 防篡改，是现代推荐选择。

## 二、非对称加密的信任假设

### 信任假设 C：私钥是保密的且只有所有者持有

- **假设内容**：RSA/ECC 私钥只有其声称的所有者持有。
- **机制来源**：非对称加密中公钥公开，私钥保密。
- **如果被突破**：私钥泄露 → 解密通信/伪造签名。

| 突破方式 | 推导的攻击 | 实际案例 |
|---------|-----------|---------|
| 私钥存储不安全 | 文件读取 → 私钥泄露 | SSH 私钥无密码保护 |
| 私钥通过侧信道泄露 | 时序/功耗分析 → 恢复私钥 | 智能卡侧信道攻击 |
| 密钥生成熵不足 | 两个用户生成了相同的密钥对 | 嵌入式设备熵不足 → 重复的 SSH 主机密钥 |
| 量子计算（未来） | Shor 算法 → 因式分解 RSA/离散对数 ECC | 后量子密码学准备 |

**推理模式：** 非对称加密的"不对称"建立在数学难题上（大数分解、离散对数）。这些难题在经典计算机上不可行，但密钥管理（生成、存储、轮换）的攻击复杂度远低于数学攻击。

### 信任假设 D：RSA 填充方案使用正确

- **假设内容**：RSA 加密使用 OAEP，签名使用 PSS。
- **机制来源**：RSA 是确定性算法——直接加密意味着相同明文产生相同密文。
- **如果被突破**：密文可被解密或伪造。

| 突破方式 | 推导的攻击 | 实际案例 |
|---------|-----------|---------|
| 无填充（教科书 RSA） | 短明文 → 枚举；相同明文 → 相同密文 | 经典 RSA 实现错误 |
| PKCS#1 v1.5 填充 | 填充预言 → 渐进解密 | Bleichenbacher 攻击 (1998) |
| e=3 且无填充 | 小指数攻击 → 开三次方根恢复明文 | 短消息 RSA |

**推理模式：** RSA 的安全性来自"填充 + 数学难题"的组合。填充不是可选的——它把确定性加密变成了概率性加密，并提供了结构验证。**使用 RSA 而不使用正确填充 = 几乎肯定不安全。**

## 三、哈希函数的信任假设

### 信任假设 E：哈希函数是单向的且无碰撞

- **假设内容**：给定哈希值 h，无法找到 x 使得 H(x)=h（抗原像性）；无法找到 x≠y 使得 H(x)=H(y)（抗碰撞性）。
- **机制来源**：哈希算法的设计目标。
- **如果被突破**：哈希值可以被逆向或碰撞。

| 突破方式 | 推导的攻击 | 实际案例 |
|---------|-----------|---------|
| MD5/SHA-1 碰撞 | 两个不同文件有相同哈希 → 签名绕过 | Flame 恶意软件使用 MD5 碰撞 |
| 长度扩展攻击（MD/SHA-1/SHA-2） | H(密钥 || 消息) 可扩展 → 伪造认证 | Flickr API 签名伪造 |
| 彩虹表 | 预计算常见密码的哈希 → 快速逆向 | 未加盐的密码哈希 |
| 哈希函数族弱点 | 碰撞成本低于预期 → 数字签名被伪造 | SHA-1 碰撞 (SHAttered) |

**推理模式：** 哈希函数的安全属性随时间退化。MD5→碰撞、SHA-1→碰撞、SHA-2→长度扩展。选择哈希函数时要考虑当前和未来一段时间的安全强度。**SHA-256 和 SHA-3 目前是安全的。HMAC 可以防御长度扩展攻击。**

### 信任假设 F：密码哈希使用慢速哈希函数

- **假设内容**：存储密码哈希时，使用了 bcrypt/scrypt/argon2 等慢速哈希。
- **机制来源**：快速哈希（SHA-256）可被 GPU/ASIC 高速计算。
- **如果被突破**：密码哈希可被快速破解。

| 突破方式 | 推导的攻击 | 实际案例 |
|---------|-----------|---------|
| 使用 SHA-256 哈希密码 | GPU 每秒数十亿次 → 快速破解 | 常见密码哈希错误 |
| 无 salt | 彩虹表 + 相同密码相同哈希 → 批量破解 | LinkedIn 密码泄露 (2012) |
| 使用 MD5 | 现代硬件每秒百亿次 → 秒破 | 各种泄露数据库 |

**推理模式：** 密码哈希 ≠ 数据完整性哈希。密码需要**慢速**、**加盐**、**内存硬**的哈希函数。这是少数开发人员理解但更少人正确实践的领域。

## 四、数字签名与证书的信任假设

### 信任假设 G：CA 系统是安全的

- **假设内容**：证书颁发机构（CA）正确验证身份后签发证书。
- **机制来源**：PKI 的信任根——浏览器/操作系统内置的根 CA。
- **如果被突破**：攻击者可以获取任意域名的有效证书。

| 突破方式 | 推导的攻击 | 实际案例 |
|---------|-----------|---------|
| CA 被攻破 | 签发任意域名证书 → MITM | DigiNotar (2011) |
| CA 签发时未充分验证 | 获得他人的域名证书 | Comodo 代理商违规 (2011) |
| 中间 CA 过度授权 | 中间 CA 可签任意域名 → 不是预期范围 | Symantec 中间 CA 违规 |
| 证书透明度 (CT) 未被强制 | 恶意证书可能不被发现 | CT 日志是实现前的状态 |

**推理模式：** PKI 的信任是"链式信任"——浏览器信任根 CA → 根 CA 信任中间 CA → 中间 CA 信任网站。链中任意一环被攻破，链下的所有信任都被破坏。

### 信任假设 H：证书验证逻辑正确

- **假设内容**：客户端正确验证证书链、过期时间、域名匹配、吊销状态。
- **机制来源**：TLS 客户端实现。
- **如果被突破**：无效证书也被接受。

| 突破方式 | 推导的攻击 | 实际案例 |
|---------|-----------|---------|
| 未验证域名是否匹配 | 证书有效但域名不匹配 → 仍接受 | 通配符证书滥用 |
| 未验证证书吊销（CRL/OCSP） | 已吊销证书仍被信任 | OCSP 装订未部署 |
| 验证时忽略中间 CA 限制 | 中间 CA 签发了超出授权的证书 | 证书链验证错误 |
| 自签名证书被信任 | 任何自签名证书 → MITM | 开发环境配置泄漏到生产 |

**推理模式：** 证书验证是一个复杂的逻辑链条（链验证 + 域名匹配 + 日期检查 + 吊销检查 + 扩展检查）。任何一个检查缺失或绕过，证书的安全性就降级到"被检查的部分"。

## 五、随机数的信任假设

### 信任假设 I：随机数生成器（CSPRNG）有足够的熵

- **假设内容**：系统能生成不可预测的随机数。
- **机制来源**：内核从硬件熵源（CPU RDRAND、硬件中断）收集熵。
- **如果被突破**：密钥/会话/一次性值可被预测。

| 突破方式 | 推导的攻击 | 实际案例 |
|---------|-----------|---------|
| 系统启动初期熵不足 | SSH 主机密钥可预测 | 嵌入式设备 SSH 指纹相同 |
| 虚拟机快照恢复 | 随机数状态回滚 → 重复的随机数 | AWS EC2 早期问题 |
| CSPRNG 状态泄露 | 知道状态 → 预测未来随机数 | 内核内存泄露 |
| 弱 PRNG（如 rand()） | 种子可预测 → 全部输出可预测 | 经典随机数漏洞 |

**推理模式：** 随机数是密码学的基石——密钥、IV、nonce、会话 ID 都依赖它。熵不足 = 所有依赖随机数的安全机制都可能是可预测的。**"随机数"不是选项，是必需品。**

## 六、协议集成攻击面

### 密码学 + 协议的组合漏洞

| 攻击 | 涉及的密码学原语 | 涉及的协议 | 推理模式 |
|------|----------------|-----------|---------|
| ZeroLogon (CVE-2020-1472) | AES-CFB8 固定 IV | Netlogon RPC | 密码学实现错误 + 协议信任 |
| POODLE (CVE-2014-3566) | CBC 模式填充验证 | SSL 3.0 | 加密模式 + 降级机制 |
| CRIME/BREACH | 压缩 + 加密 → 明文长度泄露 | HTTPS + 压缩 | 两个功能组合产生非预期信道 |
| Lucky13 | CBC 时序差异 | TLS | 实现中的时序信息泄露 |

**推理模式：** 密码学协议攻击 = 密码学算法 + 协议逻辑 + 实现的组合错误。单独看每个组件可能没问题，组合起来产生非预期的侧信道或绕过。

## 七、实战指导

```
从密码学角度评估一个系统：

1. 用了什么算法？
   → 对称：AES-GCM/ChaCha20-Poly1305 ✓
   → 非对称：RSA-2048+/ECC-256+ ✓
   → 哈希：SHA-256/SHA-3 ✓
   → 避免：DES/RC4/MD5/SHA-1 ×
   
2. 加密模式是什么？
   → AEAD (GCM/ChaCha20-Poly1305) ✓
   → CBC+HMAC 也可接受
   → ECB 或 CBC 无 MAC ×
   
3. 密钥如何管理？
   → 存储：HSM/密钥管理服务 ✓
   → 生成：CSPRNG ✓
   → 轮换：有定期轮换策略 ✓
   
4. 协议如何组合密码学？
   → 每个步骤的信任假设是什么？
   → 组合后是否有非预期行为？
   
5. 密码学实现是什么？
   → 标准库（OpenSSL/BoringSSL/rustls）✓
   → 自定义实现 ← 最高风险
```
