# 开发安全攻击面推导 — 从代码执行到漏洞

> 关联 Level 0: `dev-security/01-c-language-memory-safety.md`, `dev-security/02-rust-memory-safety.md`, `dev-security/03-common-vulnerability-patterns.md`, `dev-security/04-supply-chain-security.md`
> 关联 Level 2: Case 1 (EternalBlue/C 内存), Case 8 (Dirty Pipe/页缓存), Case 3 (XZ/供应链)
>
> 目标：枚举开发阶段的信任假设，标注假设被突破后的攻击向量。

---

## 零、开发安全的统一框架

```
开发安全 = 代码从哪里来 → 代码如何运行 → 代码如何被测试

三个阶段的信任假设:
  1. 代码来源（供应链）: 代码是可信的、未被篡改的
  2. 代码运行（执行环境）: 内存安全、类型安全、控制流完整
  3. 代码交付（发布）: 测试充分、签名有效
```

**与网络/Web/密码学攻防的不同：**
网络攻防是在"传输中"攻击，Web 是在"应用逻辑"中攻击，密码学是在"数学"中攻击。
开发安全是在"代码本身"攻击——攻击者直接针对代码的编写和运行过程。

## 一、内存安全的信任假设

### 信任假设 A：程序员不会犯错（C/C++ 核心问题）

- **假设内容**：程序员不会写出内存越界读写、use-after-free 等 bug。
- **机制来源**：无——C/C++ 没有任何内置内存安全保护。malloc-free 模型完全依赖程序员正确使用。
- **如果被突破**：攻击者控制程序内存 → 覆盖函数指针/返回地址 → 控制执行流。

| 突破方式 | 推导的攻击 | 实际案例 |
|---------|-----------|---------|
| 栈缓冲区溢出 | 覆盖返回地址 → 跳转到攻击者代码 | Stack Smashing (经典) |
| 堆缓冲区溢出 | 覆盖堆元数据 → 获得任意写原语 | EternalBlue (SMB 堆溢出) |
| 释放后使用 (UAF) | 释放后指针仍被引用 → 堆喷 + 提权 | CVE-2022-27666 (IPsec 提权) |
| 整数溢出 | 大小计算截断 → 分配小但写大 → 溢出 | 几乎所有经典漏洞 |
| 格式化字符串 | printf(user_input) → 读/写任意内存 | 训练场经典题目 |
| 空指针解引用 | 分配失败返回 NULL → 但代码继续使用 | DoS / 信息泄露 |

**缓冲区溢出攻击的通用推理链：**
```
Step 1: 找到接受用户输入的缓冲区（网络数据、文件、命令行）
Step 2: 确定是否检查了长度
Step 3: 如果没检查或检查错误 → 
  ┌─ 栈溢出: 覆盖 RET/RIP → ROP chain → 提权/RCE
  └─ 堆溢出: 覆盖堆块元数据 → 任意地址写 → 控制指针 → 提权/RCE
```

### 信任假设 B：类型转换不会出错

- **假设内容**：从大到小（int64→int32）转换时不会截断重要数据。
- **机制来源**：编译器给出警告但不会阻止。
- **如果被突破**：整数截断 → 负数 → 绕过检查。

| 突破方式 | 推导的攻击 | 实际案例 |
|---------|-----------|---------|
| signed→unsigned | -1 变成 4294967295 → 绕过大小检查 | 经典长度检查绕过 |
| large→small 截断 | 0x10000→0x0000 → 分配 0 字节但写 0x10000 | 整数溢出 → 堆溢出 |
| 负数做大小参数 | malloc(-1) → 失败返回 NULL → 但代码继续用 | UAF 变种 |

### 信任假设 C：控制流不会被劫持

- **假设内容**：程序的执行顺序由编译器决定，攻击者不能改变。
- **机制来源**：函数指针、虚函数表、返回地址保存在内存中。
- **如果被突破**：攻击者控制跳转目标 → 执行任意代码。

| 防御 | 有效性 | 绕过方式 |
|------|-------|---------|
| NX bit (数据不可执行) | 阻止在堆/栈上执行代码 | ROP: 复用已有代码 |
| ASLR | 随机化地址 → 攻击者不知道跳哪 | 信息泄露 + 暴力枚举 |
| Stack Canary | 检测栈溢出 → 提前终止 | 信息泄露 canary |
| CFI (控制流完整性) | 限制间接跳转目标 | 精细构造合法跳转 |

**ROP 的推理：**
```
不能执行新代码 (NX) → 只能跳转已有代码
问题: 已有代码中有 ret 指令后的地址
解决方案: 找 gadgets (以 ret 结尾的指令序列)
  pop rdi; ret → 设置参数
  system@plt → 执行 system("/bin/sh")
```

## 二、Rust 安全的安全假设

### 信任假设 D：unsafe 代码是安全的

- **假设内容**：unsafe 块中的代码没有违反内存安全。
- **机制来源**：unsafe 块需要程序员保证内存安全。编译器信任程序员。
- **如果被突破**：整个程序的内存安全被破坏。

| 突破方式 | 推导的攻击 |
|---------|-----------|
| unsafe 中空指针解引用 | 程序崩溃或更糟——LLVM IR 中的 `noalias` 断言 |
| unsafe 中悬垂指针 | 生命周期被绕过 → UAF |
| unsafe 中缓冲区越界 | Rust 的索引越界检查只在 safe 代码中生效 |
| FFI 调用 C 代码 | Rust 保护停止在边界 → C 代码中任何漏洞 |

**Rust 安全的真相：**
```
safe Rust → 100% 内存安全（在编译器验证范围内）
unsafe Rust → 和 C 一样脆弱，但更小范围

攻击者的策略:
  ┌─ 如果找到 Vulnerable unsafe block → 可以像 C 漏洞一样利用
  └─ 如果找不到 unsafe → 攻击类型安全（更难）
```

### 信任假设 E：类型安全保证不会被绕过

- **假设内容**：Rust 的类型系统确保变量只能以预期方式使用。
- **机制来源**：所有权、借用、生命周期系统。
- **如果被突破**：违反类型约束 → 逻辑漏洞。

| 突破方式 | 推导的攻击 |
|---------|-----------|
| `std::mem::transmute` 不安全转换 | 类型转换不检查 → 把 A 当 B 用 |
| `MaybeUninit` 未初始化读取 | 读取未初始化内存 → 信息泄露 |
| 内部可变性 (`RefCell`) 用法错误 | 运行期借用检查失败 → panic |
| `Pin` 使用错误 | 被 Pin 的数据被移动 → 违反自引用保证 |

## 三、漏洞模式的信任假设

### 信任假设 F：外部数据不会影响控制流

- **假设内容**：外部输入只影响数据，不影响程序执行路径或目标。
- **机制来源**：程序员的编码习惯。
- **如果被突破**：注入攻击、控制流劫持。

| 漏洞模式 | 信任假设 | 推导的攻击 |
|---------|---------|-----------|
| 命令注入 | "输入不会包含 shell 命令" | `input; rm -rf /` |
| SQL 注入 | "输入不会改变 SQL 结构" | `input' OR 1=1 --` |
| 路径遍历 | "输入不会包含 .." | `../../etc/passwd` |
| XSS | "输入不会包含 HTML/JS" | `<script>alert(1)</script>` |
| SSTI | "输入不会包含模板语法" | `{{7*7}}` |

**这些漏洞的统一模式：**
```
用户数据 → 被拼接到"代码"上下文中 → 数据变成代码执行
这就是"注入攻击"的本质

推理链:
  1. 找到"拼接"的位置
  2. 检查使用的上下文（SQL/Shell/HTML/模板）
  3. 检查是否有转义
  4. 如果转义不完整 → 使用上下文分隔符注入
```

### 信任假设 G：面向切面的安全控制不会被绕过

- **假设内容**：安全控制（认证、鉴权、访问控制）在正确的位置执行。
- **机制来源**：框架、中间件、过滤器。
- **如果被突破**：绕过安全控制直接访问资源。

| 突破方式 | 推导的攻击 |
|---------|-----------|
| 直接 URL 访问 | 认证只在登录页面 → 直接访问 /admin/（路径漏洞）|
| 内部 IP 访问 | SSRF 绕过 → 127.0.0.1 绕过外部防火墙 |
| 参数修改 | 后端只验证 Web 层 → 直接调用 API 修改参数 |
| 页面源文件 | 注释中的 API 路径 → 未授权访问 |

## 四、供应链安全的信任假设

### 信任假设 H：依赖的第三方代码是安全的

- **假设内容**：npm/pip/maven 仓库中的包没有恶意代码且没有漏洞。
- **机制来源**：无——包管理器不验证内容安全。
- **如果被突破**：攻击者控制依赖 → 注入后门、窃取凭据。

| 攻击方式 | 推导的攻击 | 实际案例 |
|---------|-----------|---------|
| 包名拼写错误 (typosquatting) | `requrests` vs `requests` → 安装恶意包 | Pypi 每年清理数百个 |
| 依赖混淆 | 私仓中存在的包名在公仓中也有同名→ 安装公仓恶意版本 | ua-parser-js 事件 |
| 合法包被攻破 | 维护者账号泄露 → 发布恶意更新 | event-stream (2018) |
| 合法包被社区攻破 | 开源维护者被社会工程 → 长期渗透 | XZ 后门 (2024) |
| 依赖链中的漏洞 | 间接依赖有漏洞 → 直接依赖不更新 | Equifax (Struts2) |
| 编译环境被攻破 | 编译工具链被植入后门 | SolarWinds (2020) |

**XZ 后门案例的推理链分析：**
```
Step 1: 攻击者花 2 年建立社区信任
  → 提交合法代码 → 参与维护 → 获得核心权限

Step 2: 通过 OSS-Fuzz 提代码覆盖测试
  → 添加测试用"特殊输入"→ 这些"特殊输入"触发后门

Step 3: 后门不在源代码中
  → 在 tarball 的 Makefile 中添加测试文件 → 测试文件包含编译时注入
  → `if ... ; then ... fi` 花括号后插入恶意代码
  → 后门只在发行 tarball 中有，Git 中没有

Step 4: 后门功能
  → 拦截 sshd 验证 → 如果收到特制签名 → 允许免密登录
  → 不记录日志 → 不可追踪

关键: 攻击者没有攻击"代码逻辑"
而是攻击了"社区信任过程"
```

### 信任假设 I：构建和发布工具可信

- **假设内容**：编译器和构建工具不会插入后门。
- **机制来源**：信任编译工具链本身。
- **如果被突破**：所有使用该工具构建的软件都可能被植入后门。

| 攻击方式 | 推导的攻击 |
|---------|-----------|
| 编译器后门 (Ken Thompson 反思) | 编译器在编译特定程序（如 login）时插入后门 |
| 恶意 CI/CD 插件 | Jenkins 插件 → 构建过程中泄露凭据 |
| 构建脚本修改 | Makefile/Build.xml 被篡改 → 构建插入恶意代码 |

**Ken Thompson 的反思 (Turing Award 1984)：**
```
Step 1: 修改 C 编译器 → 在编译 login 时插入后门
Step 2: 修改 C 编译器 → 检测到自己在编译 C 编译器时插入 Step 1 的修改
Step 3: 从源码中删除所有修改痕迹 → 重新编译编译器
Step 4: 新编译器编译时自动插入后门 + 自我复制代码
Step 5: 原始修改痕迹在源码中完全消失

这意味着: 信任构建工具本身是一个不证自明的假设
```

## 六、推理引擎用的攻击知识

### 代码审计的推理流程

```
给定一段代码（或知道使用的语言/框架）:

1. 确定语言/内存模型:
   C/C++ → 立即检查: 缓冲区、指针、整数
   Java/.NET/Go → 检查: 反序列化、反射、命令执行
   Rust → 检查: unsafe 块、FFI
   Python → 检查: eval/exec、反序列化、命令执行
   JS/Node → 检查: prototype pollution、eval、注入

2. 确定输入边界:
   用户输入从哪里进来? (HTTP/文件/网络/环境变量)
   输入在哪里被使用? (拼接/传给系统命令/反射调用)

3. 检查完整性保护:
   有输入验证吗? (长度/类型/格式)
   有输出编码吗? (防止 XSS/注入)
   有认证/鉴权吗? (谁可以做什么)

4. 检查依赖:
   版本有没有已知漏洞?
   依赖链深不深? (容易有间接依赖漏洞)
   包来自可信源吗?

5. 检查配置:
   默认配置安全吗?
   敏感信息硬编码了吗?
   日志记录泄露敏感信息吗?
```

### 从漏洞模式到攻击路径

```
┌─ 内存安全漏洞 (C/C++) ───────────────────────┐
│  栈溢出 → 覆盖返回地址 → ROP → 执行系统调用    │
│  堆溢出 → 覆盖函数指针 → 劫持控制流 → shell     │
│  UAF → 堆喷 → 类型混淆 → 任意代码执行           │
│  整数溢出 → 小分配大写 → 堆溢出                  │
└────────────────────────────────────────────────┘

┌─ Web 逻辑漏洞 ───────────────────────────────┐
│  SQLI → 读数据库 → 用户凭证 → 登录后台         │
│  XSS → 管理员 Cookie → 后台操作 → 提权          │
│  SSRF → 内网探测 → 内网服务漏洞 → 横向移动      │
│  路径遍历 → 读密钥文件 → SSH/RDP 登录           │
└────────────────────────────────────────────────┘

┌─ 供应链漏洞 ───────────────────────────────┐
│  恶意依赖 → 窃取凭据 → 提交后门代码             │
│  漏洞依赖 → 远程利用表面扩大                     │
│  构建工具后门 → 所有构建产物被感染               │
└────────────────────────────────────────────────┘
```

**核心洞察：** 开发安全漏洞的根源往往是「语言的抽象泄漏」。C 的抽象泄漏是内存——程序员以为自己在管理内存，实际上 malloc-free 模型要求极高纪律性。Rust 的抽象泄漏是 unsafe 和 FFI——编译器安全只覆盖到 safe 边界。Python/JS 的抽象泄漏是动态类型——运行时类型错误导致逻辑漏洞。Web 注入的抽象泄漏是数据/代码不分——用户输入既是数据也可能是代码。
