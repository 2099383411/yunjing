## 一、同源策略 (SOP) 的信任假设

### 信任假设 A：不同源的文档应该完全隔离

- **假设内容**：文档 A 不能读取文档 B 的内容，除非同源（协议+域名+端口）。
- **机制来源**：浏览器 SOP 实现——不同源时，读取操作被阻断。
- **如果被突破**：攻击者网站可以读取受害者网站的内容。

| 突破方式 | 推导的攻击 | 实际案例 |
|---------|-----------|---------|
| 同源定义过于宽松（忽略端口） | 同一域名不同端口可互读 | 旧浏览器端口劫持 |
| 子域名被攻击 → 可写父域 Cookie | 子域名 XSS → 父域 Cookie 污染 | 子域名接管 |
| 浏览器扩展/插件绕过 SOP | 跨域读取任意数据 | 恶意扩展 |
| DNS 重组攻击 → 突破同源 | 域名解析到攻击者 IP 但浏览器认为同源 | DNS 劫持 + SOP 绕过 |

**推理模式：** SOP 的"源"定义基于字符串匹配。任何能影响协议/域名/端口字符串的因素（DNS、代理、URL 解析差异）都可能绕过 SOP。SOP 的安全 = 浏览器正确实现 + DNS 正确解析 + 网络不被篡改。

### 信任假设 B：嵌入（embedding）是安全的

- **假设内容**：<img>, <script>, <iframe>, <link> 等标签加载跨域资源是安全的。
- **机制来源**：SOP 允许嵌入但不允许读取——嵌入不泄露内容。
- **如果被突破**：攻击者可以通过嵌入触发副作用 → 利用受害者的身份。

| 突破方式 | 推导的攻击 | 实际案例 |
|---------|-----------|---------|
| <script> 加载任意 URL → 执行上下文 | JSONP 回调 → 数据泄露 | JSONP 劫持 |
| <img> 标签触发 GET 请求 | CSRF 攻击 → 在受害者身份下执行操作 | 经典 CSRF |
| <form> 提交到任意 URL | CSRF POST → 状态变更 | 各种 CSRF 变种 |
| <iframe> 加载目标网站 | 点击劫持 → 受害者无意识操作 | 点击劫持 (Clickjacking) |

**推理模式：** SOP 允许"嵌入"但禁止"读取"。但嵌入本身就产生了副作用——GET 请求携带 Cookie、脚本在目标上下文中执行。防御思路：区分"无副作用的嵌入"（img）和"有副作用的嵌入"（form）。

## 二、跨域资源共享 (CORS) 的信任假设

### 信任假设 C：服务器端 CORS 配置是正确的

- **假设内容**：Access-Control-Allow-Origin 等头部精确控制了哪些源可以跨域读取。
- **机制来源**：服务器通过 HTTP 头部声明跨域策略。
- **如果被突破**：不该被访问的资源被跨域读取。

| 突破方式 | 推导的攻击 | 实际案例 |
|---------|-----------|---------|
| Access-Control-Allow-Origin: * | 任意网站可读取响应 → 数据泄露 | 公开 API 的私有数据 |
| Origin 回显（动态 ACAO） | 攻击者的 Origin 被反射 → 允许读取 | CORS 配置错误扫描 |
| 信任的 Origin 包含子域名漏洞 | 子域名 XSS → 跨域读取主域名数据 | 子域名攻击链 |
| 预检请求（OPTIONS）未正确限制 | 非简单请求的误判 | 预检绕过 |

**推理模式：** CORS 的安全边界由服务器控制。服务器说"谁可以读"——如果配置过于宽松（* 或反射 Origin），SOP 的保护就被关闭了。CORS 配置 = 白名单管理，任何白名单中的源被攻破 = CORS 保护失效。

## 三、内容安全策略 (CSP) 的信任假设

### 信任假设 D：CSP 能有效限制可执行的内容

- **假设内容**：CSP 头部限制了哪些来源的脚本/样式可以执行。
- **机制来源**：浏览器在加载资源前检查 CSP 策略。
- **如果被突破**：CSP 被绕过 → XSS 仍然可行。

| 突破方式 | 推导的攻击 | 实际案例 |
|---------|-----------|---------|
| script-src 包含 CDN/jsonp | 利用信任的 CDN 上的可操控端点 → 执行任意 JS | CSP 绕过 via JSONP/angular |
| 非严格 CSP (允许 'unsafe-inline') | 内联脚本仍可执行 → XSS 有效 | 宽松 CSP |
| 通过 <base> 标签劫持相对路径 | 相对路径的脚本被替换 | CSP + base 标签绕过 |
| JSONP 端点未验证回调名 | 回调名可包含 JS 代码 → CSP 信任的源执行 | Google API JSONP 绕过 |

**推理模式：** CSP 的信任模型是白名单——信任某些源。但信任的源如果有"可攻击的端点"（JSONP API、上传点），白名单就成了绕过后门。**CSP 的安全 = 白名单中所有端点的安全。**

## 四、HTTP Cookie 的信任假设

### 信任假设 E：Cookie 是安全的身份标识

- **假设内容**：浏览器正确管理 Cookie 的发送范围（Domain/Path/Secure/HttpOnly）。
- **机制来源**：Cookie 头部自动附加到匹配的请求。
- **如果被突破**：Cookie 被窃取或滥用。

| 突破方式 | 推导的攻击 | 实际案例 |
|---------|-----------|---------|
| Cookie 未标记 Secure → 明文传输 | HTTP 侧网络嗅探 → 会话劫持 | 公共 WiFi 会话窃取 |
| Cookie 未标记 HttpOnly → JS 可读 | XSS → document.cookie 读取 → 会话劫持 | 经典 XSS 会话窃取 |
| Domain 设置过于宽泛（.example.com） | 任意子域名 XSS → 读取父域 Cookie | 子域名攻击 |
| Path 限制可被绕过 | 同域名其他路径可访问 Cookie | Cookie path 绕过 |
| SameSite 未设置 | CSRF 请求携带 Cookie | 跨站请求伪造 |

**推理模式：** Cookie 是"自动附加"的身份凭证。自动附加 = 任何能触发请求的攻击（XSS/CSRF）都能利用它。防御思路：缩小 Cookie 的作用范围（Secure+HttpOnly+SameSite+Limited Domain）。

## 五、认证机制的信任假设

### 信任假设 F：登录后的会话标识不可伪造

- **假设内容**：会话 ID (session token) 随机且不可预测。
- **机制来源**：服务器生成随机 token，与用户会话绑定。
- **如果被突破**：攻击者可以预测或窃取会话 ID。

| 突破方式 | 推导的攻击 | 实际案例 |
|---------|-----------|---------|
| 会话 ID 使用时间戳/MD5(用户名) | 预测会话 ID → 任意用户登录 | 经典会话预测 |
| 会话 ID 在 URL/Referer 中泄露 | 第三方获取会话 → 劫持 | 会话泄露 |
| 登录后未更新会话 ID | 会话固定 → 攻击者诱使用户使用已知 ID | 会话固定 (Session Fixation) |
| JWT 未验证签名 | 伪造任意 JWT → 冒充用户 | JWT none 算法攻击 |

**推理模式：** 会话安全 = 会话 ID 的随机性 × 传输安全 × 存储安全。三者中最弱的一环决定整体安全。

### 信任假设 G：OAuth 2.0 流程各参数验证正确

- **假设内容**：redirect_uri、state、client_secret 等参数被正确验证。
- **机制来源**：OAuth 2.0 授权码流程。
- **如果被突破**：授权码被窃取 → 用户账号被绑定到攻击者。

| 突破方式 | 推导的攻击 | 实际案例 |
|---------|-----------|---------|
| redirect_uri 验证不严格（允许子路径） | 授权码发送到攻击者的回调 | OAuth 隐式授权劫持 |
| state 参数缺失或固定 | CSRF 攻击 → 攻击者的账号绑定用户身份 | OAuth 账号绑定攻击 |
| client_secret 泄露 | 使用泄露的凭据获取 token | OAuth 凭据泄露 |

**推理模式：** OAuth 是授权委托协议——用户授权应用访问资源。每个参数的验证缺失都可能导致授权被劫持。**redirect_uri 是最关键的参数**——它决定了授权码去往何处。

## 六、跨协议/跨层攻击面

### Web + OS 交叉

| 攻击面 | 推导 | 案例 |
|-------|------|------|
| file:// 协议访问本地文件 | 浏览器允许 file:// 访问本地系统 | 本地文件泄露 |
| WebRTC 绕过 VPN | STUN 请求暴露真实 IP | WebRTC IP 泄露 |
| Service Worker 劫持 | 注册恶意 SW → 控制所有加载的页面 | SW 劫持 |

### Web + 网络交叉

| 攻击面 | 推导 | 案例 |
|-------|------|------|
| Host 头攻击 | 后端使用 Host 头构建链接 → 缓存投毒/密码重置 | Host 头注入 |
| X-Forwarded-For 伪造 | 后端信任 XFF → IP 白名单绕过 | XFF 伪造 |
| WebSocket 未验证 Origin | 跨站 WebSocket 连接 → CSWSH | WebSocket 劫持 |

### Web + 密码学交叉

| 攻击面 | 推导 | 案例 |
|-------|------|------|
| HTTPS 降级 | 攻击者拦截 HTTPS → 降级到 HTTP → 明文传输 | SSLStrip |
| 混合内容 | HTTPS 页面加载 HTTP 资源 → 脚本被篡改 | 混合内容攻击 |

## 七、实战指导

```
从 Web 安全角度评估一个 Web 应用：

1. 认证如何工作？
   → Cookie / JWT / OAuth / SAML
   → 每个机制的信任假设是什么？
   → 会话 ID 可预测吗？JWT 签名验证了吗？

2. 跨域策略是什么？
   → CORS 配置是否过于宽松？
   → CSP 是否允许 unsafe-inline？
   → SOP 是否有可绕过点？

3. 用户输入的流向？
   → 存储型/反射型 XSS 可能吗？
   → CSRF 保护是否存在？
   → 输入 → 数据库 → 输出 链条中每步的安全？

4. 第三方依赖？
   → 使用了什么 CDN/库？
   → CDN 被攻破 → 脚本被替换 → 所有用户受影响
```
