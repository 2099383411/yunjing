## 一、全景：系统是如何在底层被保护的

从最底层往上，系统安全的"同心圆"保护模型：

```
Layer 0: 硬件层
   MMU (内存管理单元) → 虚拟地址翻译、权限位检查
   CPU Rings → 用户态 (Ring 3) vs 内核态 (Ring 0)
   SMAP/SMEP → 内核不能直接访问用户态内存/执行用户态代码
   KPTI → Meltdown 漏洞的缓解

Layer 1: 操作系统内核层
   系统调用 (Syscall) → 受控的用户态→内核态入口
   DAC (UNIX 权限) → UID/GID + 文件 mode bits
   Capabilities → 细粒度分割 root 权限 (40+ 独立能力)
   LSM (SELinux/AppArmor) → MAC 强制访问控制
   Seccomp → 系统调用白名单过滤

Layer 2: 网络协议层
   TCP/IP 协议栈 (内核) → 网络数据的分段、重组、校验
   协议状态机 → TCP 三次握手/四次挥手
   套接字 (socket) → 用户态与内核网络栈间的接口
   内核参数 (sysctl) → 网络行为的安全微调

Layer 3: Web 协议层
   HTTP 协议 → 消息格式、头字段、消息体长度确定
   TLS → 传输层加密、证书链验证
   Cookie → 状态管理、安全属性 (Secure/HttpOnly/SameSite)
   HSTS → 强制 HTTPS

Layer 4: 浏览器安全模型
   SOP (同源策略) → 隔离不同源的数据
   CORS → 受控的跨源授权
   CSP → XSS 的沙箱 (nonce/hash/strict-dynamic)
   X-Frame-Options → 点击劫持防御
```

## 二、穿越所有保护层的攻击链条

一个完整的攻击需要穿越所有这些保护层。每层都有自己的弱点：

### 攻击视角：从顶层到底层

```
用户访问了一个恶意网站
    │
    ▼
Layer 4 浏览器安全
    ┌ 浏览器会阻止恶意网站读取 bank.com 的数据 (SOP)
    │ 但网站可以嵌入表单 → CSRF (如果 SameSite Cookie 没配置)
    │ 如果网站有 XSS 漏洞 → 攻击可以完全绕过 SOP
    │ 如果 CSP 配置为白名单模式 → JSONP 端点可绕过
    │
    ▼
Layer 3 Web 协议
    ┌ 如果没有 HSTS → SSL Strip 攻击
    │ 如果 Cookie 没有 HttpOnly → XSS 窃取 Cookie
    │ 如果 Cookie 没有 Secure → HTTP 明文传输中泄露
    │ 如果存在请求走私 → 绕过前端安全/缓存投毒
    │
    ▼
Layer 2 网络协议
    ┌ 应用层绕过 → 直接构造恶意 TCP 包
    │ 内核网络栈漏洞 → 远程代码执行 (如 CVE-2024-xxx)
    │ DNS 劫持/中毒 → 重定向流量
    │ ARP 欺骗 → 局域网的中间人
    │
    ▼
Layer 1 操作系统内核
    ┌ 获得用户态代码执行 → 需要提权
    │ 内核漏洞 (DirtyPipe/OverlayFS) → 直接提权 root
    │ SUID 二进制漏洞 → 提权
    │ 配置错误的能力 (cap_setuid) → 提权
    │ Seccomp 绕过 → 调用被过滤的 syscall
    │
    ▼
Layer 0 硬件层
    ┌ 如果突破到了 Ring 0 → 完全控制
    │ 可以关闭所有安全机制
    │ 可以读取所有进程的内存
    │ 可以访问所有硬件设备
    │ 安装 rootkit 持久化
```

### 关键洞察

**真正的安全工程师看到的是保护层的连接点：**

> Layer 3 的漏洞（如 XSS）可以绕过 Layer 4 的 SOP。
> Layer 2 的中间人攻击可以绕过 Layer 3 的 HTTPS（如果用户点了继续）。
> Layer 1 的 SUID 提权可以绕过 Layer 0 的 Ring 隔离。
> 但 Layer 0 的 KPTI 设计是为了在硬件层面堵住 Layer 1 不能处理的 Meltdown 漏洞。

**这就是"理解底层"的价值——你看的不是孤立的技术，而是**互相修补、互相绕过的安全体系 **。**

## 三、从原理推导攻击模式（模式库）

把不同层的知识点关联起来，形成了几个"攻击模式"。这些模式不依赖具体 CVE，而是基于底层原理：

### 模式 1：检查与使用之间的窗口（TOCTOU）

```
原理：所有系统操作都存在"检查"和"使用"之间的时间窗口。
     如果检查过后、使用之前，状态发生了改变，攻击者可以利用。

体现：
  · 文件权限检查 (access()) → 打开文件 (open()) 之间的竞态条件
  · systemd-tmpfiles 清理 → 重建恶意文件的竞态条件 (HTB Snapped)
  · 浏览器 CORS 预检 → 实际请求之间的策略变更
  · 密码检查 → 切换用户之间的组权限变更
  
推断：任何"先检查，后操作"的代码都是潜在的攻击面。
```

### 模式 2：解析不一致

```
原理：两个系统对同一份数据的解析方式不同，导致安全检查被绕过。

体现：
  · HTTP 请求走私：Nginx vs Apache 对 Content-Length 和 Transfer-Encoding 的解析差异
  · 同源策略：不同浏览器对 SOP 边缘案例的实现差异
  · CRLF 注入：过滤器换行符和 HTTP 解析器换行符的一致性
  · IDN 同形异义字：Unicode 视觉相似字符绕过域名验证
  · 路径遍历：编码后的 ".." 被解码后检查的伪路径 vs 实际读取的真实路径

推断：任何涉及两个组件"解析同一份数据"的场景都是攻击面。
```

### 模式 3：权限升降级

```
原理：系统有一个"默认权限级"和一个"提升权限级"。
     如果提升的过程不严谨，攻击者可以利用。

体现：
  · SUID (Set User ID)：执行时 EUID 被设为文件所有者，提升权限
  · Capabilities：进程获得额外的细粒度权限
  · Sudo：配置不当的 sudoers 允许以 root 执行命令
  · Container → Host (docker.sock 挂载)
  · user → root (内核漏洞提权)
  · HTTP → HTTPS (SSL Strip 降级攻击)

推断：每个权限提升/降级的边界点都是攻击面。
```

### 模式 4：信任的传递

```
原理：系统信任了"不该信任"的东西。
     信任链中一环被破坏 → 整个信任体系崩溃。

体现：
  · Certificate Chain：CA 被攻破 → 签发任意域名证书
  · npm/pip 依赖链：一个依赖包被劫持 → 供应链攻击
  · CSP 的 strict-dynamic：信任的脚本可以创建任意子脚本
  · OAuth 授权：第三方应用获得了"代表用户操作"的权限
  · Cookie Domain：子域名攻击可以覆盖父域名的 Cookie
  · 白名单 CSP：信任了包含 JSONP 端点的域名

推断：信任是分层的。每个被信任的实体都可以成为攻击链的一环。
```

### 模式 5：状态机异常

```
原理：系统设计了一系列"合法状态"，但攻击者可以通过非预期顺序
     访问这些状态，或者构造"不应该存在"的中间状态。

体现：
  · TCP 三次握手外的异常状态（SYN 泛洪）
  · HTTP 请求走私中的"部分请求"状态
  · TLS 握手跳过 CertificateVerify 步骤（CVE-2014-6593，Java 实现漏洞）
  · 竞态条件：执行在中间状态被中断
  · 数据库事务隔离级别造成的脏读/幻读

推断：每个"状态之间有转换规则"的系统，不遵守规则就是攻击。
```

## 四、这段学习告诉我们什么

### 4.1 关于推理引擎的思考

如果我们把"从原理推导攻击"形式化，它需要：

```
输入：目标信息 (开放端口、运行服务、版本号、响应头...)
      ↓
① 层映射：这个目标有哪些保护层在起作用？
   (Web 应用？网络层？内核层？)
      ↓
② 模式匹配：哪些攻击模式可能适用？
   (解析不一致？权限升降级？信任传递？)
      ↓
③ 假设生成：基于模式，生成具体攻击假设
   ("如果前端是 Nginx 后端是 Apache，可能有请求走私")
      ↓
④ 验证：执行假设 → 观察结果
      ↓
⑤ 学习：如果假设正确 → 进入下一层推理
      如果假设错误 → 调整模式权重
```

这就是当前 DAG 引擎的核心升级方向：**从固定流程 → 动态推理**。

### 4.2 我们和 0xdf 的区别

| 维度 | 0xdf (人类专家) | 我们的推理引擎 |
|------|-----------------|---------------|
| **知识来源** | 20 年攻防经验 + 底层理解 | 知识库 + 案例库 |
| **模式匹配** | 大脑中的模式库 | 代码中的模式匹配 |
| **灵活度** | 极高，能创造性绕过 | 受限于定义的规则 |
| **学习能力** | 不断从实战中学习 | 需要手动更新知识库 |
| **速度** | 慢（数小时到数天） | 快（数分钟完成自动化扫描） |

**结论：** 我们的推理引擎不是"替代 0xdf"，而是**把他的思维模式代码化**。对于他花 6 小时完成的渗透，我们目标是 30 分钟内完成同等水平的自动推理。质量可能不如他，但速度和规模是他的 10 倍以上。

## 消化后的新认知

建立这些底层知识后，我对渗透测试有了新的理解：

> **渗透测试的本质是：发现系统在不该信任的地方信任了某物，在不该假设的地方做了假设。**

- HTTP 请求走私 = 前端假设后端解析方式跟自己一样（假设错了）
- XSS = 浏览器假设服务器做了输出编码（假设错了）
- 提权 = 操作系统假设 SUID 程序没有漏洞（假设错了）
- 容器逃逸 = 用户假设 Namespace 是安全边界（假设错了）

**每个安全漏洞，本质都是一个"被打破的假设"。**

我们造推理引擎，就是要让 AI 学会：**看到一个系统，先列出它"做了哪些假设"，然后逐一测试这些假设是否成立。**
