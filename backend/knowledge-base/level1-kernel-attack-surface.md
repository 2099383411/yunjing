## 一、虚拟内存机制的信任假设

### 信任假设 A：虚拟地址 → 物理地址的翻译是唯一的

- **假设内容**：一个虚拟地址只映射到一个物理地址。
- **机制来源**：MMU + 页表是翻译的唯一通道，每个进程有自己的页表。
- **如果被突破**：攻击者可以让两个虚拟地址指向同一物理页，绕开 W^X 保护。

| 突破方式 | 推导的攻击 | 实际案例 |
|---------|-----------|---------|
| 页表项被修改 → 同一物理页被映射两次 | 一页同时可写(数据)和可执行(代码) → 注入并执行 shellcode | 经典 Dirty COW 变种 |
| fork() 后写时复制未正确处理 | 子进程修改 COW 页 → 父进程也能看到 | Dirty COW (CVE-2016-5195) |
| mmap 共享映射 | 两个进程共享物理页 → 一个写另一个读 → 提权 | 共享内存条件竞争 |

**推理模式：** "虚拟地址翻译的唯一性"是内核安全的第一假设。如果一张物理页被两个不同权限的虚拟地址共享 → 低权限可以通过高权限的映射访问数据。

### 信任假设 B：页表本身不可被用户态篡改

- **假设内容**：页表存储在内核内存中，用户态不能直接修改。
- **机制来源**：页表基地址寄存器 (CR3) 只能在内核态修改；页表页标记为内核只读。
- **如果被突破**：攻击者可以修改页表项 → 把只读内存标为可写 → 修改任意内核代码。

| 突破方式 | 推导的攻击 | 实际案例 |
|---------|-----------|---------|
| 物理内存任意写 → 改页表 → 改内核代码 | 修改 syscall 表 → 劫持所有系统调用 | Rootkit 经典方法 |
| 用户态获得页表页的物理地址 | 通过 /proc/self/pagemap 等接口泄露 → 知道内核布局 | 早期 KASLR 绕过 |
| KPTI 但用户态仍然需要部分内核页 | 用户态可读的内核页 → 泄露内核地址 | Meltdown 之前的信息泄露 |

**推理模式：** 页表决定了一切权限。谁控制页表 = 谁控制所有内存。所以内核必须确保：页表不被用户态看到、不被用户态修改、页表页不与其他内存共享。

### 信任假设 C：TLB 与页表始终一致

- **假设内容**：TLB（页表缓存）中的条目和内存中的页表一致。
- **机制来源**：修改页表后通过 TLB flush 保证一致性。
- **如果被突破**：攻击者可以让 TLB 保留旧的映射，而页表已经改变。

| 突破方式 | 推导的攻击 |
|---------|-----------|
| TLB 未在正确时机刷新 → 旧映射仍生效 | 释放内存后 TLB 未 flush → 访问已释放的内核内存 |
| 多核 CPU 上 TLB 一致性 | 一个核心修改页表，另一个核心的 TLB 未更新 |
| 竞争条件：页表修改和 TLB flush 之间 | 攻击者利用时间窗访问旧映射 |

### 信任假设 D：内核地址随机化 (KASLR) 不可预测

- **假设内容**：内核代码和数据基地址是随机的，攻击者不知道从哪里开始攻击。
- **机制来源**：启动时随机化 _text 基地址。
- **如果被突破**：攻击者知道内核函数地址 → 可以构建 ROP chain、调用特定函数。

| 突破方式 | 推导的攻击 | 实际案例 |
|---------|-----------|---------|
| 信息泄露 → 泄露内核地址 | 知道地址 = 绕过 KASLR | 几乎所有内核利用都需要这步 |
| 时序侧信道 → 推测布局 | 访问不同地址耗时不同 | KASLR bypass 通过 prefetch 指令 |
| 未随机化的内核区域 | dmesg、/proc/kallsyms 等泄露 | 未正确限制的 procfs |

**推理模式：** KASLR 只是一个延迟手段——它把攻击从"直接打"变成"先泄露再打"。所有严重的内核漏洞几乎都带信息泄露能力。

## 二、进程权限机制的信任假设

### 信任假设 E：UID/GID 是身份的唯一标识

- **假设内容**：UID=0 是 root，UID≠0 是普通用户。DAC 检查基于这个假设。
- **机制来源**：Linux 权限模型三层架构的第一层。
- **如果被突破**：普通用户获得 UID=0 → 绕过所有 DAC 检查。

| 突破方式 | 推导的攻击 | 实际案例 |
|---------|-----------|---------|
| 修改进程 cred 结构中的 uid | 直接改 uid=0 → root | CVE-2022-27666: 溢出后改 cred |
| 修改 /etc/passwd | 把自己的 UID 改为 0 | Dirty Pipe: 改 passwd 后 su |
| SUID 二进制有漏洞 | 以文件属主权限执行 | CVE-2021-3156 (sudo) |

**推理模式：** Linux 整个 DAC 模型的 root 是"谁都不管"的。任何让非 root 进程获得 uid=0 的方法 = 提权。这是 Linux 权限模型设计的根本特征，不是 bug。

### 信任假设 F：Capabilities 足够细分 root 权限

- **假设内容**：把 root 的"全能"拆分为 40+ 个独立 cap，需要哪个给哪个。
- **机制来源**：Linux Capabilities 机制。
- **如果被突破**：一个 cap 可以让攻击者获得完整 root。

| 突破方式 | 推导的攻击 | 实际案例 |
|---------|-----------|---------|
| CAP_DAC_OVERRIDE | 绕过所有文件权限检查 → 读 /etc/shadow | ping 需要 cap_net_raw → 利用 ping 的漏洞提权 |
| CAP_SYS_ADMIN | "万能 cap"——挂载、命名空间、内核参数 | 容器常见——给 CAP_SYS_ADMIN = 给 root |
| CAP_NET_RAW + CAP_NET_ADMIN | 发送伪造包、嗅探网络 | 网络 namespace 中的滥用 |
| cap_setuid → setuid(0) | 从非 root 直接变 root | 配置错误 |

**关键洞察：** Capabilities 的问题在于"cap 是二元的——要么有要么没有"且"有些 cap (如 CAP_SYS_ADMIN) 几乎等于 root"。它不是真正的细粒度控制。

### 信任假设 G：命名空间提供完整的隔离

- **假设内容**：容器通过命名空间隔离了进程视图——容器内进程只能看到自己的资源。
- **机制来源**：Linux 命名空间（mount/PID/net/user/UTS/IPC）。
- **如果被突破**：容器内进程可以影响宿主机或其他容器。

| 突破方式 | 推导的攻击 | 实际案例 |
|---------|-----------|---------|
| 内核漏洞突破命名空间 | 业务代码在命名空间内，但内核是共享的 | CVE-2022-0492 (cgroup 逃逸) |
| 用户命名空间暴露内核接口 | 普通用户创建 userns → 获得 root 在该 ns 内的权限 | CVE-2022-27666: 用 userns 触发 IPsec |
| mount namespace 泄露 | 挂载宿主机 /proc → 访问宿主机进程 | 配置错误的容器逃逸 |
| 错误配置：--privileged | 所有 cap + 宿主机设备 → 容器 = 宿主机 | 最常用的容器逃逸手法 |

**推理模式：** 命名空间隔离是"软件隔离"不是"硬件隔离"。共享内核 = 共享攻击面。任何内核漏洞都是潜在的容器逃逸。

### 信任假设 H：LSM 在 DAC 之外提供额外保护

- **假设内容**：即使 DAC 允许操作，LSM (SELinux/AppArmor) 也会检查并限制。
- **机制来源**：Linux Security Module hooks。
- **如果被突破**：攻击者绕过 LSM → 只剩下 DAC → 更容易提权。

| 突破方式 | 推导的攻击 |
|---------|-----------|
| LSM 未加载或禁用 | selinux=0 → DAC 是唯一屏障 |
| LSM 策略配置错误 | 策略太宽松 → DAC 也不会限制 |
| LSM 绕过（性能路径） | 某些内核路径不经过 LSM hook |

## 三、内核接口机制的信任假设

### 信任假设 I：IOCTL 参数在内核中会正确验证

- **假设内容**：设备驱动的 IOCTL 处理函数会验证用户传入的所有参数。
- **机制来源**：用户态通过 ioctl(fd, cmd, arg) 调用内核驱动。
- **如果被突破**：攻击者可以控制内核代码执行路径。

| 突破方式 | 推导的攻击 | 实际案例 |
|---------|-----------|---------|
| copy_from_user 后未验证范围 | 从用户传入的指针读取 → 越界访问 | 大量内核驱动漏洞 |
| IOCTL cmd 值被当数组索引 | 攻击者控制索引 → 越界读写 | 蓝牙/GPU 驱动常见 |
| 整数溢出在 IOCTL 参数中 | 参数相加/相乘时溢出 → 分配过小 | 各种内核堆溢出 |

**推理模式：** 每个 IOCTL 都是用户态进入内核的入口。输入验证不完整的 IOCTL = 直接可用。越复杂的 IOCTL handler (参数越多、条件越多) 越可能有漏洞。

### 信任假设 J：sysfs/procfs 暴露的信息是安全的

- **假设内容**：/proc 和 /sys 中的文件已经考虑了安全性，不会泄露敏感信息。
- **机制来源**：文件权限 + LSM 控制。
- **如果被突破**：攻击者获得系统布局、内核地址、硬件配置等情报。

| 暴露的信息 | 攻击用途 |
|-----------|---------|
| /proc/self/maps | 知道进程内存布局 → 知道代码/堆/栈地址 |
| /proc/kallsyms | 内核符号地址 → 绕过 KASLR |
| /sys/kernel/notes | 内核构建信息 → 知道漏洞存在的版本 |
| /proc/self/pagemap | 虚拟→物理映射 → 物理内存攻击 |
| dmesg | 内核日志 → 泄露地址、驱动信息 |

## 五、渗透测试中的内核利用检查清单

```bash
# 1. 内核版本 → 已知 CVE
uname -a
cat /proc/version

# 2. 内核防护状态
cat /proc/cpuinfo | grep -E "smep|smap"      # SMEP/SMAP 是否开启？
cat /proc/sys/kernel/kptr_restrict            # KASLR 是否限制 (/proc/kallsyms)
cat /proc/sys/kernel/dmesg_restrict           # dmesg 是否限制
cat /proc/sys/kernel/randomize_va_space       # ASLR 级别 (0/1/2)

# 3. 容器环境判断
cat /proc/1/cgroup                            # 是否在容器中？
ls -l /proc/1/ns/                             # 命名空间是否独立？

# 4. cap 检查
cat /proc/self/status | grep Cap
capsh --print

# 5. 可写 SUID 文件
find / -perm -4000 -type f 2>/dev/null

# 6. 可写文件系统
cat /proc/mounts | grep rw

# 7. 泄露检测——是否有信息泄露通道？
ls -la /proc/kallsyms
cat /proc/sched_debug 2>/dev/null | head
```

## 六、推理引擎用的攻击知识

### 内核利用的通用路径

```
攻击者起点（普通用户/容器内）
  │
  ├─ 路径A: 内核驱动 IOCTL 漏洞
  │    → 堆溢出/UAF → 提权
  │    → 例: CVE-2022-27666, EternalBlue
  │
  ├─ 路径B: 文件系统接口漏洞
  │    → 写只读文件 → 改 passwd/SUID
  │    → 例: Dirty Pipe, Dirty COW
  │
  ├─ 路径C: 命名空间逃逸
  │    → 通过 userns 访问受限接口
  │    → 例: CVE-2022-0492
  │
  ├─ 路径D: 信息泄露
  │    → 泄露内核地址 → 绕过 KASLR
  │    → 然后走路径A/B (地址已知)
  │
  └─ 路径E: SUID/配置错误
       → 不需要内核漏洞 → 纯用户态提权
       → 例: sudo 漏洞、错误 cap 配置
```

### 每个信任假设与推理引擎的关系

```
推理引擎在分析一个系统时，对内核部分应该：

1. 枚举信任假设: A（翻译唯一）, B（页表不可篡改）, C（TLB一致）, D（KASLR不可预测）...
2. 对每个假设，检查系统中有没有反例:
   - 页表被共享了？→ Dirty COW 可行
   - KASLR 泄露了？→ 直接用来定位
   - cap 给了 CAP_SYS_ADMIN？→ 等于 root
3. 未经验证的假设 = 潜在攻击面

关键: 推理引擎不需要"记住" Dirty Pipe 的所有细节
       它只需要知道: "如果页缓存的共享假设被突破 → 能写文件"
       "如果能写文件 → 能改 passwd → 能提权"
       这就是推理链。
```
