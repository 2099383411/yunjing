# 操作系统内存管理与安全

> 知识库层级：Level 1 - 操作系统内核基础
> 日期：2026-06-04
> 来源：Linux Kernel Docs, OS Course Notes (Cornell CS3410), GeeksforGeeks

## 一、核心概念：虚拟内存与 MMU

### 1.1 为什么要有虚拟内存？

虚拟内存是现代操作系统的核心抽象。它给每个进程一个"假象"——好像自己独占整个连续的内存空间。这解决了三个问题：

| 问题 | 虚拟内存的解决方式 |
|------|-------------------|
| 多进程冲突 | 每个进程有自己的虚拟地址空间，互不干扰 |
| 内存碎片 | 物理内存可以不连续，虚拟地址空间看起来连续 |
| 内存不足 | 可以将不常用的内容换出到磁盘（swap） |

### 1.2 MMU（内存管理单元）

MMU 是 CPU 内部的硬件组件，负责把虚拟地址翻译成物理地址。

**关键机制：**

```
CPU → 发出虚拟地址 → MMU → 查页表（TLB缓存） → 物理地址 → 访问内存
                         ↓ 如果页不存在
                         触发缺页中断（Page Fault）→ 内核处理
```

- MMU 是硬件（在 CPU 里），不是软件
- 页表是内核管理的数据结构（存在物理内存里），MMU 读取页表
- TLB（Translation Lookaside Buffer）是 MMU 内部的缓存，加速地址翻译
- **进程切换时，OS 更换页表指针寄存器**，所以一个进程看不到另一个进程的内存

### 1.3 页表条目（PTE）的关键位

每个页表条目包含了远比地址映射更多的信息：

| 字段 | 含义 | 安全意义 |
|------|------|----------|
| Present Bit | 页面是否在物理内存中 | 访问不存在的页 → 段错误/缺页 |
| Read/Write | 是否可写 | 代码段设为只读 → 阻止代码注入修改 |
| User/Supervisor | 用户态还是内核态可访问 | 用户态代码无法读写内核内存 |
| Execute/NX | 是否可执行 | NX bit → 阻止在数据段执行代码 |
| Dirty Bit | 页面是否被写过 | 用于页面置换算法 |
| Accessed Bit | 页面是否被访问过 | 用于页面置换算法 |

**安全关键：** User/Supervisor 位是用户态/内核态隔离的硬件基础。

## 二、安全隔离机制

### 2.1 特权级（Rings）

x86 架构支持 4 个特权级（Ring 0-3），现代操作系统只用两个：

| 级别 | 名称 | 运行内容 | MMU 权限 |
|------|------|----------|----------|
| Ring 0 | 内核态/超级用户态 | 操作系统内核、驱动 | 可访问 User+Supervisor 页 |
| Ring 3 | 用户态 | 应用程序 | 只能访问 User 标记的页 |

**关键点：** 用户态代码无法直接访问标记为 Supervisor 的页。即使知道物理地址，MMU 也会拒绝访问。

### 2.2 系统调用（Syscall）

用户态程序需要执行特权操作（读写文件、创建进程等）时，通过系统调用进入内核态。

**流程：**
```
用户态 (Ring 3)                   内核态 (Ring 0)
    │                                  │
    │  syscall 指令                     │
    │ ──────────────────────────────→  │
    │  1. CPU 切换到 Ring 0            │
    │  2. 切换到内核栈                 │
    │  3. 保存用户态寄存器             │
    │  4. 查 sys_call_table[nr]        │
    │  5. 执行内核函数                 │
    │                                  │
    │  ←────────────────────────────── │
    │  sysret 指令                     │
    │  1. 恢复用户态寄存器             │
    │  2. 切换回 Ring 3                │
```

**安全关键：**
- sys_call_table 是一个函数指针数组，syscall number 是索引
- 如果对 syscall number 不做边界检查 → **CVE-2026-31483**（2026年，s390 架构的安全漏洞）
- seccomp-bpf 可以在系统调用执行前做过滤（浏览器、容器核心用这个）

### 2.3 KPTI（内核页表隔离）

**背景：** Meltdown 漏洞（2018年）利用了 CPU 的推测执行——即使访问被拒绝（User/Supervisor 位拦截），CPU 在推测执行中仍然会加载数据到缓存，攻击者可以通过时序侧信道读取内核内存。

**解决方案：** 维护两套页表
- **用户态页表：** 只映射必要的内核入口（用于处理 syscall 和中断）
- **内核态页表：** 完整映射

```
传统方案 (无 KPTI):
用户态页表包含全部内核映射 (只是标记为 Supervisor)
→ Meltdown 漏洞利用推测执行绕过检查

KPTI 方案:
用户态页表完全不包含内核内存
→ 推测执行时没有内核数据可加载
→ 但每次 syscall 需要切换 CR3 寄存器（性能损失）
```

### 2.4 SMAP / SMEP

| 机制 | 全称 | 作用 |
|------|------|------|
| SMEP | Supervisor Mode Execution Prevention | 禁止内核执行用户页面的代码 |
| SMAP | Supervisor Mode Access Prevention | 禁止内核直接访问用户页面的数据 |

**安全意义：** 如果内核有一个漏洞（如缓冲区溢出），攻击者不能直接跳转到用户态准备的 shellcode（SMEP），也不能直接读写用户态数据篡改系统调用参数（SMAP）。

## 三、进程隔离

### 3.1 进程地址空间

每个进程拥有独立的虚拟地址空间。在 Linux 上，典型的 64 位进程布局：

```
0x0000000000000000 ┌─────────────────────┐
                   │ 代码段 (.text)      │ ← 只读 + 可执行
                   │ 数据段 (.data)      │ ← 读写
                   │ BSS 段 (.bss)       │ ← 读写
                   │ 堆 (heap)           │ ← 运行时分配
                   │       ↓             │
                   │       ↑             │
                   │ 栈 (stack)          │ ← 线程栈
                   │ 环境变量/参数       │
0x00007FFFFFFFF000 ┝━━━━━━━━━━━━━━━━━━━┥ ← 用户态/内核态分界
                   │ 内核空间            │ ← 仅内核可访问
0xFFFFFFFFFFFFFFFF └─────────────────────┘
```

**ASLR（地址空间布局随机化）：** 每次程序启动时，栈、堆、动态库的基址随机化。防止攻击者预知地址后构造 ROP 链。

### 3.2 任务结构体（task_struct）

内核中，每个进程对应一个 `task_struct`，包含：

```c
struct task_struct {
    struct cred *cred;       // 进程凭证（UID, GID, capabilities）
    struct mm_struct *mm;    // 内存管理信息（页表指针）
    struct fs_struct *fs;    // 文件系统信息
    struct files_struct *files; // 打开的文件描述符表
    struct nsproxy *nsproxy; // 命名空间
    // ...
};
```

**安全关键：** 权级提升（Privilege Escalation）的核心目标就是修改当前进程的 `cred` 结构，把自己的 UID 改成 0（root）。

## 四、安全相关的漏洞类别

### 4.1 内核漏洞类型

| 类型 | 原理 | 危害 |
|------|------|------|
| 缓冲区溢出 | 内核代码没有做边界检查 | 覆盖内核内存 → 代码执行 |
| Use-After-Free | 释放后继续使用指针 | 利用悬空指针 → 代码执行 |
| 整数溢出 | 整型运算超出范围 | 导致分配过小的缓冲区 |
| 条件竞争 (Race) | 操作不具有原子性 | 检查和使用之间的窗口 |
| 未初始化数据 | 使用未初始化的内核变量 | 信息泄露 |
| 逻辑错误 | 权限检查缺失/有缺陷 | 绕过安全限制 |

### 4.2 从用户态到内核的漏洞利用链

```
用户态触发 → 传递恶意参数给系统调用
           → 内核函数处理时出错（如缓冲区溢出）
           → 攻击者控制内核执行流
           → 修改当前进程的 cred
           → 提权到 root
           → 逃逸容器/访问敏感数据
```

**真实案例：Copy Fail 漏洞**
- 利用 `AF_ALG` socket + `splice()` 系统调用的组合
- 攻击者打开一个特权命令的 fd
- 通过 `splice()` 将数据分块送入内核加密 API
- 利用内核内存处理缺陷 → 覆盖 page cache 中的 `/usr/bin/su`
- 其他进程执行 `su` 时 → 攻击者的代码以 root 权限运行
- **关键点：** 不需要执行 shellcode，只通过系统调用的特殊组合就能完成攻击

### 4.3 容器逃逸本质

容器和宿主机共享同一个内核。区别在于：
- Namespace → 进程"看到"的隔离（文件系统、网络、PID）
- Cgroup → 资源限制（CPU、内存）
- Seccomp → 系统调用白名单

**逃逸路径：**
```
容器内 → 找到一个内核漏洞 → 触发
       → 获得宿主机内核权限
       → 突破 namespace 隔离
       → 访问宿主机文件系统/进程
```

## 五、理解这个层面对渗透测试的意义

为什么理解 MMU 和系统调用对渗透测试很重要？

| 你看到的现象 | 底层原理 | 攻击思路 |
|-------------|----------|----------|
| 程序崩溃（Segmentation Fault） | 访问了无效虚拟地址，MMU 触发缺页，内核发送 SIGSEGV | 缓冲区溢出测试 |
| 提权漏洞需要本地执行 | 需要先进入用户态，然后才能调用 syscall 触发内核漏洞 | 先获取初始 shell 再找提权 |
| 容器内执行 mount 失败 | seccomp 过滤了 mount 系统调用 | 检查 seccomp 配置 |
| ASLR 绕过难度大 | 每次启动随机化地址空间 | 需要信息泄露配合 |
| NX bit 阻止 shellcode 执行 | 页表标记数据段不可执行 | 改用 ROP 链 |
