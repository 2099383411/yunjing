## 一、C 语言内存安全的信任假设

### 信任假设 A：缓冲区有足够的空间

- **假设内容**：写入缓冲区的数据不会超过其分配的大小。
- **机制来源**：C 语言不自动检查边界，由程序员确保不越界。
- **如果被突破**：缓冲区溢出 → 覆盖相邻内存 → 控制流劫持。

| 突破方式 | 推导的攻击 | 实际案例 |
|---------|-----------|---------|
| 栈缓冲区溢出 | 覆盖返回地址 → ROP | 经典栈溢出（如 HTTP 头解析） |
| 堆缓冲区溢出 | 覆盖堆元数据 → 任意写原语 | EternalBlue 堆溢出 |
| 整数溢出导致分配过小 | 分配 N 字节但写 M 字节（M>N） | EternalBlue 整数截断 |
| off-by-one 错误 | 多写一个字节到相邻内存 | 各类 off-by-one 漏洞 |
| 格式化字符串误用 | printf(user_input) → 栈读取/写入 | 经典格式化字符串漏洞 |

**推理模式：** "缓冲区有足够空间"是 C 语言中最基本的假设，也是最常被违反的。每个数据拷贝操作（memcpy/strcpy/sprintf）都需要证明"源 ≤ 目标"。**任何没有长度检查的拷贝操作 = 潜在的溢出。**

### 信任假设 B：已释放的内存不会被访问

- **假设内容**：free() 后的指针不再使用（Use-After-Free 不发生）。
- **机制来源**：free() 释放内存，但指针中的地址不变。
- **如果被突破**：悬空指针 → 访问已释放内存 → 任意代码执行。

| 突破方式 | 推导的攻击 | 实际案例 |
|---------|-----------|---------|
| 释放后未置 NULL | 悬空指针被解引用 → 访问已释放堆块 | 浏览器/UAF 经典漏洞 |
| 同一内存被重新分配后旧指针仍在使用 | 旧指针操作新对象的恶意数据 | 堆 UAF 提权 |
| 竞争条件中释放和使用的时序差 | 一个线程释放，另一个线程使用 | 内核 UAF (CVE-2024 系列) |

**推理模式：** UAF 的核心问题是"指针生命周期 < 对象生命周期"。解决思路：要么确保指针在对象释放后不可用（置 NULL），要么让对象生命周期 > 所有引用（引用计数/GC）。

### 信任假设 C：整数运算不会溢出/截断

- **假设内容**：整数加减乘除的结果在类型范围内。
- **机制来源**：C 语言的整数类型有固定宽度（int=32bit, short=16bit 等）。
- **如果被突破**：计算结果错误 → 分配不足/逻辑绕过。

| 突破方式 | 推导的攻击 | 实际案例 |
|---------|-----------|---------|
| 加法溢出 | a+b < a → 绕过大小检查 | 分配 size 检查绕过 |
| 乘法溢出 | a*b 超出范围 → 分配过小 | 数组分配大小计算错误 |
| 符号错误 | 有符号→无符号转换 → 负值变超大正数 | memcpy 负长度绕过 |
| 截断（DWORD→WORD） | 大值截断后变小 → 分配不足 | EternalBlue: DWORD→WORD 截断 |

**推理模式：** 整数溢出不是算术问题，是**信任问题**——代码信任"用户提供的数字不会导致计算错误"。任何"用数值计算后再分配"的代码路径都需要检查溢出。

## 二、Rust 安全的信任假设

### 信任假设 D：Unsafe 代码块是安全的

- **假设内容**：unsafe 块内的操作（解引用裸指针、调用外部函数）不会导致内存不安全。
- **机制来源**：Rust 的安全保证只在 Safe 代码中由编译器保证，Unsafe 依赖程序员。
- **如果被突破**：Unsafe 中的错误 → 绕过了 Rust 的安全保证。

| 突破方式 | 推导的攻击 | 实际案例 |
|---------|-----------|---------|
| unsafe 中解引用悬空指针 | UAF → 任意读/写 | Rust 标准库历史漏洞 |
| unsafe 中创建错误的生命周期 | 引用比数据活得久 → UAF | 自定义数据结构 |
| FFI 调用 C 库 → C 库有漏洞 | C 库的漏洞进入 Rust 代码 | 通过 FFI 引入内存错误 |
| unsafe 中违反别名规则 | 两个可变引用指向同一内存 → 数据竞争 | UnsafeCell 误用 |

**推理模式：** Rust 的安全模型把"不安全"隔离到 unsafe 块中。但 unsafe 块的数量和复杂度决定了 Rust 代码的实际安全水平。**Rust 项目的安全 ≈ unsafe 代码的安全。**

### 信任假设 E：Send/Sync 标记正确

- **假设内容**：类型正确标注了 Send（可跨线程传递）和 Sync（可跨线程共享）。
- **机制来源**：Rust 的线程安全通过 Send/Sync trait 标记。
- **如果被突破**：非线程安全的类型被跨线程使用 → 数据竞争。

| 突破方式 | 推导的攻击 | 实际案例 |
|---------|-----------|---------|
| 错误标注 Send | 非线程安全类型跨线程 → UAF/数据竞争 | 自定义类型未检查线程安全 |
| 内部可变性 + 非 Sync | RefCell 跨线程 → 运行时 panic/UB | 误用 RefCell |
| 错误实现 Arc 内的不安全类型 | Arc 中的非 Send 类型在 Drop 时竞争 | 自定义 Drop 实现 |

**推理模式：** Send/Sync 是 Rust 线程安全的"类型级别契约"。错误标注 = 编译器认为安全但实际上不安全。

## 三、通用漏洞模式的信任假设

### 信任假设 F：用户输入不会破坏程序逻辑

- **假设内容**：用户输入被正确处理，不会产生非预期效果。
- **机制来源**：程序从外部获取输入（HTTP 参数、文件、环境变量等）。
- **如果被突破**：注入攻击 → 数据泄露/命令执行。

| 突破方式 | 推导的攻击 | 实际案例 |
|---------|-----------|---------|
| 输入被拼接到 SQL 查询 | SQL 注入 → 数据库任意操作 | 经典 SQL 注入 |
| 输入被拼接到 HTML | XSS → 执行任意 JS | 存储型/反射型 XSS |
| 输入被拼接到命令 | 命令注入 → 任意命令执行 | Shellshock |
| 输入被拼接到文件路径 | 路径遍历 → 读取任意文件 | ../ 路径穿越 |
| 输入被拼接到模板 | SSTI → 模板引擎 RCE | Server-Side Template Injection |
| 输入被反序列化 | 反序列化 → 任意对象创建 | Java/Python 反序列化 |
| 输入被拼接到 JNDI 查询 | JNDI 注入 → 远程加载恶意类 | Log4Shell |

**推理模式：** 注入攻击 = 输入数据变成了代码/逻辑的一部分。核心问题是**数据和控制流的分离**——当用户输入被当做代码解析时，就产生了注入。防御思路：总是把输入当做数据（参数化查询/转义/沙箱），而不是拼接到上下文中。

### 信任假设 G：反序列化的数据是可信的

- **假设内容**：从网络/文件读取的序列化数据不会被篡改或构造。
- **机制来源**：序列化/反序列化机制从字节流重建对象。
- **如果被突破**：攻击者构造恶意序列化数据 → 任意对象创建 → RCE。

| 突破方式 | 推导的攻击 | 实际案例 |
|---------|-----------|---------|
| Java 反序列化调用任意类 | 利用 gadget chain → RCE | Apache Commons Collections |
| Python pickle 反序列化 | __reduce__ 执行任意代码 | pickle 反序列化 RCE |
| PHP 反序列化 | 魔术方法滥用 | Drupal/WordPress 漏洞 |
| .NET 反序列化 | TypeNameHandling=Auto | ysoserial.net |

**推理模式：** 反序列化 = 让攻击者决定"程序在反序列化过程中调用哪些方法"。如果这些方法可以进一步操作对象，就形成了 gadget chain。**即使反序列化后验证数据，反序列化过程中的方法调用也可能已经造成了破坏。**

## 四、供应链的信任假设

### 信任假设 H：开源包是善意且安全的

- **假设内容**：公开包管理器（npm/PyPI/RubyGems/Crates.io）上的包是安全的。
- **机制来源**：开发者相信包管理器维护了包的安全性。
- **如果被突破**：恶意包进入依赖树 → 代码执行。

| 突破方式 | 推导的攻击 | 实际案例 |
|---------|-----------|---------|
| 依赖混淆（私有包名被注册到公开源） | 构建系统优先使用公开源 → 恶意包 | Alex Birsan 漏洞 ($130K) |
| 拼写错误（typosquatting） | 开发者敲错包名 → 下载恶意包 | npm 常见 typosquatting |
| 维护者账号被攻破 | 恶意版本更新 → 恶意代码 | event-stream (2018) |
| 长期贡献后门（社会工程学） | 获得信任后植入后门 | XZ Utils (2024) |
| 构建系统被攻破 | 合法代码 + 恶意构建产物 | SolarWinds (2020) |

**推理模式：** 供应链攻击利用了**信任的传递性**。开发者信任包管理器 → 包管理器信任维护者 → 维护者的身份可以被窃取或伪造。信任链中的任意一环被突破 → 整个依赖树被污染。

### 信任假设 I：构建过程是可复现和可审计的

- **假设内容**：分发版本的代码和仓库中的代码一致。
- **机制来源**：开发者相信构建服务器产生了正确的产物。
- **如果被突破**：构建产物包含源代码中没有的恶意代码。

| 突破方式 | 推导的攻击 | 实际案例 |
|---------|-----------|---------|
| 构建服务器被攻破 | 构建产物被篡改 | SolarWinds |
| 编译时注入（测试数据隐藏后门） | 源码看起来正常，构建时执行隐藏脚本 | XZ Utils |
| 依赖镜像被污染 | 镜像中的依赖被替换 | pip/npm 镜像投毒 |

**推理模式：** 源代码安全 ≠ 构建产物安全。可复现构建（Reproducible Builds）和 SBOM（软件物料清单）是解决这个问题的方向——但部署率极低。

## 五、跨层攻击面

### 开发安全 + OS 内核

| 攻击面 | 推导 | 案例 |
|-------|------|------|
| 堆溢出 → 堆元数据篡改 → 任意写 → cred 修改 | 用户态漏洞 → 内核权限提升 | 几乎所有内核提权 |
| 栈溢出 → ROP → 绕过 KASLR/SMEP/SMAP | 用户态漏洞 → 绕过内核防御 | 现代内核利用 |
| UAF → 内核对象劫持 → 权限提升 | 内核漏洞利用 | Dirty Pipe 变种 |

### 开发安全 + 密码学

| 攻击面 | 推导 | 案例 |
|-------|------|------|
| 时序侧信道 → 密钥恢复 | 分支条件/内存访问模式依赖秘密数据 | Spectre v1 |
| 缓存侧信道 → 跨进程数据泄露 | CPU 缓存状态依赖访问模式 | Meltdown/Spectre |
| 内存泄露 → 密钥/随机数泄露 | 越界读导致密钥材料泄露 | Heartbleed |

### 开发安全 + 供应链

| 攻击面 | 推导 | 案例 |
|-------|------|------|
| 依赖中的漏洞 → 间接引入 | 不是自己写的漏洞，却在代码中执行 | Log4j 供应链 |
| 构建工具链漏洞 → 所有产物受影响 | 编译器/链接器被篡改 | XZ 的编译时注入思想 |
| 镜像/容器基础镜像漏洞 | 基础 OS 包中的漏洞进入所有容器 | 容器供应链 |

## 六、实战指导

```
从开发安全角度评估一个代码库：

1. 使用什么语言？
   → C/C++: 内存安全由程序员确保 → 检查所有边界
   → Rust: 安全由编译器保证 → 检查 unsafe 块
   → Java/Go/Python: 内存安全 → 检查注入/反序列化/逻辑

2. 输入来源和流向？
   → 用户 → API → 数据库 → 模板 → 用户
   → 每一步是否有验证/转义/参数化？

3. 依赖管理？
   → 依赖数量？间接依赖数？
   → 是否有 SBOM？是否定期扫描 CVE？
   → 构建是否可复现？

4. 信任边界？
   → 哪些代码信任了什么假设？
   → 用户命名空间 = 不可信
   → 内核态 = 可信但可能有漏洞
   → Socket 输入 = 不可信
```
