# Level 1：攻击面推导总结

> 目标：从全部 16 份文档的底层原理出发，推导所有可能的攻击方向。
> 核心问题：**这个机制是怎么工作的？如果绕过或破坏它，攻击者能得到什么？**
> 
> 这比 CVE 列表更强——CVE 只告诉你已知的漏洞，
> 这些推导告诉你**系统设计上注定的脆弱点**。

---

## 一、OS 内核层攻击面

### 1.1 虚拟内存 → 攻击面

**基础原理回顾：**
- CPU 通过 MMU 把虚拟地址翻译为物理地址
- 页表是这层翻译的权威数据源
- 页表位于内存中，由操作系统管理

**可推导的攻击方向：**

| # | 攻击方向 | 推导逻辑 | 实际攻击技术 |
|---|---------|---------|------------|
| 1 | **双重映射** | 同一物理页映射到两个虚拟地址 → 一个写代码，一个写数据 → 绕过 W^X | MapWritable + Memcpy → MapExecutable（经典 shellcode 注入） |
| 2 | **页表篡改** | 如果能写页表项 → 把只读页标为可写 → 修改代码 | Dirty Cow (CVE-2016-5195): 写时复制未锁定 page 导致可写私有映射 |
| 3 | **TLB 不一致** | TLB 是页表的缓存 → 如果 TLB 未刷新 → 旧映射仍生效 | TLB poisoning，多核 TLB 竞争 |
| 4 | **KASLR 绕过** | 内核地址随机化 → 但通过侧信道获取偏移 → 定位关键结构 | Meltdown 读内核内存、KASLR bypass 通过时序/缓存 |
| 5 | **物理内存读取** | /dev/mem, /dev/kmem, 过时设备节点暴露物理内存 | 传统 rootkit 直接读写物理内存 |
| 6 | **Meltdown/Spectre** | CPU 推测执行访问了越权内存 → 通过缓存侧信道泄露 | Meltdown: 读内核内存；Spectre: 越界数组访问 |
| 7 | **用户态页表泄露 (KPTI)** | 即使 KPTI 隔离了大部分，仍有必要的内核页在用户态可见 | 通过暴露的内核地址推测布局 |

**核心洞察：** 虚拟内存的所有攻击都围绕一个矛盾——「CPU 需要查表翻译地址，但表本身也是内存中的数据」。谁控制了表，谁就控制了内存。

### 1.2 进程权限模型 → 攻击面

**基础原理回顾：**
- 进程有 UID/GID → DAC 检查
- root (uid=0) 绕过所有 DAC 检查
- Capabilities 细分 root 权限
- LSM 在 DAC 之后做额外检查

**可推导的攻击方向：**

| # | 攻击方向 | 推导逻辑 | 实际攻击技术 |
|---|---------|---------|------------|
| 1 | **UID 0 劫持** | DAC 允许 uid=0 做任何事 → 只要获得 uid=0 → 无限制访问 | SUID 提权、cap_setuid、setuid(0) |
| 2 | **Capability 泄露** | 进程继承父进程 capabilities → 子进程可能获得不必要的特权 | Ping 命令的 cap_net_raw → 利用 ping 漏洞提权 |
| 3 | **Capability 与 DAC 冲突** | 文件有 644 权限，但进程有 CAP_DAC_OVERRIDE → 绕过文件权限 | 滥用 CAP_DAC_READ_SEARCH 读系统文件 |
| 4 | **LSM 绕过** | LSM hooks 在内核中 → 如果 LSM 未加载或加载错误 → DAC 是唯一屏障 | 启动时传递 selinux=0 |
| 5 | **命名空间逃逸** | 容器通过 namespace 隔离进程视图 → 但内核是共享的 → 内核漏洞 = 容器逃逸 | CVE-2022-0492 (cgroup 逃逸)、CVE-2016-5195 (Dirty Cow 逃逸) |
| 6 | **SUID 滥用** | SUID 位让普通用户以文件属主权限执行 → 属主是 root → 用户获得 root | 所有 SUID 提权漏洞（如 CVE-2021-3156 sudo 漏洞） |

**核心洞察：** Linux 权限模型是「层叠的过滤器」——DAC → Capabilities → LSM。但上层可以绕过下层，下层是上层的 fallback。理解这个层级关系就知道哪些组合可以逃脱。

## 二、网络协议栈攻击面

### 2.1 TCP/IP 栈 → 攻击面

**基础原理回顾：**
- sk_buff 是网络数据的内核表示
- 数据包从网卡 → 协议栈 → socket
- 内核参数控制 TCP 行为

**可推导的攻击方向：**

| # | 攻击方向 | 推导逻辑 | 实际攻击技术 |
|---|---------|---------|------------|
| 1 | **TCP 序列号预测** | TCP 依赖序列号标识有效包 → 能预测序列号 → 伪造 TCP 包 | TCP 会话劫持 |
| 2 | **TCP RST 攻击** | 伪造 RST 包且序列号在窗口内 → 关闭连接 | RST 攻击断开连接 |
| 3 | **SYN Flood** | TCP 三次握手要求服务器为 SYN 分配资源 → 大量 SYN 耗尽资源 | 经典的 DoS 攻击 |
| 4 | **IP 分片重组攻击** | IP 分片在接收端重组 → 重叠偏移导致覆盖数据 | 分片重叠绕过防火墙、Teardrop 攻击 |
| 5 | **sk_buff 越界** | 内核网络栈处理 sk_buff 时需小心边界 → 否则越界读写 | 内核网络驱动/RDS 协议内存损坏 |
| 6 | **参数旁路** | net.ipv4.conf.all.rp_filter 可绕过 → 源地址欺骗 | IP spoofing + 反射攻击 |

### 2.2 HTTP/1.1 → 攻击面

**基础原理回顾：**
- 请求行 → 首部 → 空行 → 正文
- Content-Length 和 Transfer-Encoding 决定消息边界
- 请求走私利用前端/后端对边界理解的差异

**可推导的攻击方向：**

| # | 攻击方向 | 推导逻辑 | 实际攻击技术 |
|---|---------|---------|------------|
| 1 | **请求走私** | 前端代理和后端对请求边界解析不同 → 一个请求被当作两个 | CL.TE/TE.CL/TE.TE 请求走私 |
| 2 | **响应走私** | 同理 → 后端的响应被前端误解边界 → 缓存投毒 | Web 缓存投毒 |
| 3 | **Host 头攻击** | Host 头被用于路由和 URL 生成 → 攻击者控制该头 → 密码重置到恶意服务器 | Host header injection |
| 4 | **Cookie 属性缺失** | Cookie 有 Secure/HttpOnly/SameSite 属性 → 属性缺失 = 攻击面 | Session hijacking via XSS |
| 5 | **不安全的重定向** | 3xx 响应 + Location 头 → 如果攻击者控制 Location → 钓鱼 | Open redirect |

**核心洞察：** HTTP 的问题在「消息边界解析不一致」。前端、后端、WAF、CDN 都按自己的方式解析同一个请求——只要有歧义，就有漏洞。

### 2.3 TLS → 攻击面

已经在 crypto 文档中全面分析了。TL;DR：
- 降级攻击（POODLE/FREAK/LOGJAM/DROWN）
- 填充预言攻击（Bleichenbacher/Lucky13）
- 证书验证不完整（不检查 CN/SAN、过期、自签名）
- 弱密码套件（RC4/3DES/CBC/EXPORT）
- 密钥交换无前向保密（RSA 密钥交换）

### 2.4 DNS → 攻击面

**基础原理回顾：**
- DNS 是层次化的分布式数据库
- 查询从根 → TLD → 权威
- UDP 53 (快但不安全)
- Transaction ID (16-bit) 是唯一的认证

**可推导的攻击方向：**

| # | 攻击方向 | 推导逻辑 | 实际攻击技术 |
|---|---------|---------|------------|
| 1 | **ID 碰撞** | 只有 16-bit TID (65536 种可能) → 暴力猜解 | DNS 缓存投毒（经典 Kaminsky 攻击） |
| 2 | **UDP 源端口固定** | 旧 DNS 解析器使用固定源端口 → 攻击者只需猜 TID | 早期 BIND 漏洞 |
| 3 | **递归解析** | 解析器代查上游 → 缓存不利响应 → 毒害 | Kaminsky: 注入 NS 记录指向攻击者服务器 |
| 4 | **DNSSEC 未部署** | 没有密码学签名验证 → 任何人都可冒充 DNS 响应 | 中间人 DNS 欺骗 |
| 5 | **域名劫持** | 域名注册信息过时 → 转移域名 → 控制邮箱 | 域名劫持社交工程 |
| 6 | **通配符 DNS** | *.example.com → 任何不存在的子域名指向同一 IP | 子域名枚举泄露内部服务 |
| 7 | **DNS 隧道** | DNS 查询允许任意域名文本 → 编码数据在其中 → C2 通信 | dns2tcp, Iodine 隧道 |

**核心洞察：** DNS 的安全假设是「只有 16-bit TID 是认证凭证」——就这么点。没有 DNSSEC 的 DNS 就是一个信任裸奔的协议。

### 2.5 SSH → 攻击面

**基础原理回顾：**
- 密钥交换 (DH/ECDH) → 建立加密通道
- 公钥认证 (RSA/Ed25519)
- 端口转发、Agent 转发

**可推导的攻击方向：**

| # | 攻击方向 | 推导逻辑 | 实际攻击技术 |
|---|---------|---------|------------|
| 1 | **暴力破解** | 密码认证是交互式 → 没有速率限制 → 无限尝试 | 字典攻击 SSH（常见云服务器被扫） |
| 2 | **弱密钥** | DSA 密钥 1024-bit → 可被分解 | DSA 1024 经不起 NSA 级别的攻击 |
| 3 | **私钥泄露** | ~/.ssh/id_rsa 无密码 → 泄露即万能 | 文件泄露、Git 泄露、任意文件读取 |
| 4 | **Agent 转发滥用** | ssh -A 在跳板机上暴露 Agent socket → A 从 B 到 C 的转发 | SSH 跳板机提权横向移动 |
| 5 | **known_hosts 固定** | 首次连接验证 host key，存入 known_hosts → 有人伪造 host key 首次连接的 MITM | SSH 首次连接 MITM（未启用 CA） |
| 6 | **端口转发逃逸** | -R 把内部端口暴露到外部 → 跳板机成为内网入口 | SSH 反向隧道逃逸出容器 |

**核心洞察：** SSH 本身加密是安全的。问题主要在配置（允许密码、允许 root 登录、弱密钥）和认证管理（私钥泄露、Agent 转发滥用）。

### 2.6 SMB → 攻击面

**基础原理回顾：**
- SMB 是 Windows 文件/打印机共享协议
- NTLM 认证（挑战-响应）
- SMB2 缓解了 SMB1 的很多问题
- SMB 内核驱动直接处理复杂包结构

**可推导的攻击方向：**

| # | 攻击方向 | 推导逻辑 | 实际攻击技术 |
|---|---------|---------|------------|
| 1 | **SMB1 协议复杂 → 内存损坏** | SMB1 有大量不规范的命令和数据结构 → 解析器易越界 | EternalBlue (MS17-010): SMBv1 缓冲区溢出 |
| 2 | **NTLM 中继** | NTLM 挑战-响应不绑定目标服务器 → 把 NTLM 响应中继到另一台服务器 | SMB Relay: 拿到 DC 的 NTLM → 中继到另一台服务器 |
| 3 | **Pass-the-Hash** | NTLM hash 可以直接用于认证（不需要密码明文） | PtH: 用 hash 登录远程机器 |
| 4 | **匿名 IPC$ 访问** | IPC$ 共享默认可匿名访问 → 枚举用户、共享 | SMB null session 枚举网络 |
| 5 | **未经签名的 SMB** | SMB 签名默认不强制 → 中继攻击于中间人 | SMB 会话劫持 |
| 6 | **SMB 压缩** | SMB 压缩函数的实现不当 → 内存损坏 | 最近发现的 SMB 压缩漏洞 |

**核心洞察：** SMB 的根本问题是它太老了。SMB1 诞生于 1980 年代互联网不安全的世界。NTLM 认证设计时没考虑中继。每一个补丁都只是打补丁，不是重新设计协议。

## 三、Web 安全攻击面

### 3.1 浏览器安全模型 → 攻击面

**基础原理回顾：**
- 同源策略（SOP）：协议+端口+主机相同才可互访
- CORS：服务器声明允许哪些来源跨域
- CSP：服务器声明允许加载哪些资源

**可推导的攻击方向：**

| # | 攻击方向 | 推导逻辑 | 实际攻击技术 |
|---|---------|---------|------------|
| 1 | **CORS 配置过宽松** | Access-Control-Allow-Origin: * → 任意网站可发跨域请求 | 跨域数据窃取（API 暴露） |
| 2 | **CORS 带凭证** | Access-Control-Allow-Credentials: true + Origin 反射 → 任意网站发带 Cookie 请求 | CORS + 反射 Origin 劫持 Cookie |
| 3 | **CSP 配置不当** | CSP 只 report 不 enforce → 实际无保护；'unsafe-inline' → 允许内联脚本 | CSP bypass (JSONP, 已知CDN劫持) |
| 4 | **postMessage 无来源校验** | window.postMessage 不校验 e.origin → 任意窗口可发送消息 | postMessage XSS 注入 |
| 5 | **iframe 点击劫持** | X-Frame-Options 未设置 → 页面可被 iframe 加载 → 透明覆盖 | 点击劫持（likejacking） |
| 6 | **protocol handler 绕过** | navigator.registerProtocolHandler → 自定义协议处理 | 已注册的 mailto: → 启动邮件客户端时 URL 注入 |

### 3.2 XSS 与注入 → 攻击面

**统一注入模型：**
```
数据拼接代码时未正确转义 → 数据和代码边界模糊 → 注入

  HTML拼接:   "Hello " + username        → XSS
  SQL拼接:    "SELECT * FROM users WHERE id=" + id → SQLi
  命令行拼接:  "cat " + filename           → CMDi
  模板拼接:   {{ user_input }}             → SSTI
```

**从统一模型推导的攻击：**

| # | 注入类型 | 关键区别 | 利用方式 |
|---|---------|---------|---------|
| 1 | **DOM XSS** | 客户端 JS 将不可信数据放入 innerHTML | <img src=x onerror=alert(1)> |
| 2 | **反射 XSS** | 服务端把 URL 参数反射到 HTML | ?q=<script>alert(1)</script> |
| 3 | **存储 XSS** | 恶意数据持久化（评论/用户名） | 提交恶意评论 → 管理员查看时执行 |
| 4 | **SQL 注入** | 字符串拼接 SQL | ' OR 1=1 -- |
| 5 | **命令注入** | | ; cat /etc/passwd |
| 6 | **SSTI** | 模板引擎拼接用户数据 | {{config}}、{{7*7}} |
| 7 | **XXE** | XML 解析器未禁用外部实体 | <!ENTITY xxe SYSTEM "file:///etc/passwd"> |
| 8 | **LDAP 注入** | LDAP 查询过滤器中拼接 | )(&(uid=admin)) |

**核心洞察：** 所有注入都是同一个问题——代码/数据边界被模糊了。区别只是「在哪个上下文中拼接」。知道上下文等于知道 payload 语法。

### 3.3 认证与授权 → 攻击面

| # | 攻击方向 | 推导逻辑 | 实际攻击技术 |
|---|---------|---------|------------|
| 1 | **JWT alg 混淆** | 服务器信任 JWT header 中的 alg → 改为 none → 绕过签名验证 | JWT none 攻击 |
| 2 | **JWT 弱密钥** | HS256 密钥是密码短语 → 弱密码 → 暴力破解 | jwt_tool / hashcat 破解 |
| 3 | **JWT 密钥泄露** | RS256 公钥可获取 → 如果公钥被当作 HMAC 密钥 → 伪造 | JWT algorithm confusion (RS256→HS256) |
| 4 | **OAuth redirect_uri** | redirect_uri 校验不严格 → 钓鱼 Code 窃取 | OAuth 回调劫持 |
| 5 | **OAuth CSRF** | state 参数不使用 → 攻击者绑定受害者账户到自己的 OAuth 应用 | OAuth 账户劫持 |
| 6 | **Session 固定** | 认证后不更改 Session ID → 攻击者先种植 Session ID | Session fixation |
| 7 | **Session 可预测** | Session ID 生成基于时间/递增ID → 猜解 | Tomcat session ID 预测 |

## 四、开发安全攻击面

### 4.1 C 内存安全 → 攻击面

**基础原理回顾：**
- C 没有边界检查 → 类 == 自由 == 危险
- 栈：局部变量、返回地址、ebp
- 堆：malloc/free、chunk metadata
- 所有防御机制（ASLR/NX/Canary/PIE/RELRO）都是"补丁"

**可推导的攻击方向：**

| # | 攻击方向 | 推导逻辑 | 实际攻击技术 |
|---|---------|---------|------------|
| 1 | **栈溢出** | 局部缓冲区大小固定 → 写超过边界 → 覆盖返回地址 | Ret2libc, ROP |
| 2 | **堆溢出** | 堆块数据写入超过分配大小 → 覆盖相邻堆块 | House of Force, Fastbin Attack |
| 3 | **Use-After-Free** | free() 后指针未置空 → 释放的内存被重新分配 + 悬垂指针访问 | UAF 在浏览器/内核中极常见 |
| 4 | **Double Free** | 同一内存释放两次 → 堆管理元数据损坏 | glibc tcache dup |
| 5 | **整数溢出** | sizeof(buf) = n → 分配 n+1 字节 → 整数环绕 → 缓冲区太小 → 堆溢出 | 整数溢出 + 堆溢出组合 |
| 6 | **格式化字符串** | printf(user_input) → %x%x%n 泄露栈内容、写任意地址 | %n 写入内存 |
| 7 | **向后溢出** | 索引负数 → 写 buffer 之前的内存 | intel AEP 内存管理漏洞 |

**核心洞察：** 每个 C 漏洞的根本原因是「C 语言本身不检查内存访问的有效性」。所有防御机制都是 OS/编译器级别加的约束。找到一个绕过 = 回到原始的任意内存访问。

### 4.2 Rust 内存安全 → 局限性攻击面

| # | 攻击方向 | 推导逻辑 | 实际攻击技术 |
|---|---------|---------|------------|
| 1 | **unsafe 块漏洞** | Rust 用 unsafe 调用 C → unsafe 中 UAF/溢出 | C 库接口漏洞 |
| 2 | **逻辑漏洞** | 类型系统不检查逻辑正确性（如边界检查是否在正确位置） | Off-by-one 检查点放错位置 |
| 3 | **Panic 处理** | unwrap() 导致 panic → 服务崩溃 | DoS via controlled false input |
| 4 | **FFI 类型混淆** | 跨 C/Rust 边界时类型不匹配 | 传错误大小导致堆内存泄漏 |
| 5 | **同步原语误用** | Mutex 锁顺序固定 → 死锁或竞争 | 条件竞争 |

### 4.3 漏洞统一模式 → 攻击面

| 模式 | 原理 | 表现形式 |
|------|------|---------|
| **未经验证的信任** | 系统假设输入是安全的（但实际不是） | x 所有注入、SSRF、路径遍历、反序列化 |
| **缺失边界检查** | 处理固定大小缓冲区时不检查 size | x 所有缓冲区溢出 |
| **时序竞争** | 操作不是原子的 → 中间状态可被干扰 | TOCTOU（文件检查→使用） |
| **状态不一致** | 系统预期状态 A 但实际是 B | 两步认证、交易重放、反序列化类型混淆 |
| **回退到不安全路径** | 安全功能失败时回退到不安全模式 | Fail-open 认证、DNSSEC 回退到普通 DNS |
| **信息泄露** | 错误消息包含内部状态 | 用户枚举（"用户不存在" vs "密码错误"） |

### 4.4 供应链安全 → 攻击面

| # | 攻击方向 | 推导逻辑 | 实际攻击技术 |
|---|---------|---------|------------|
| 1 | **依赖混淆** | 内部包名在公共注册表不存在 → 任何人可注册 → 自动安装 | Birsan 攻击（PayPal/Apple/MS 被拿） |
| 2 | **Typosquatting** | 发行者误输包名 → 恶意包被安装 | reqeusts (typo of requests) |
| 3 | **维护者劫持** | 包维护者账户被攻破 → 发布恶意更新 | event-stream、UA-Parser-JS |
| 4 | **构建投毒** | CI/CD 环境被攻破 → 构建产物包含后门 | SolarWinds（最严重后果） |
| 5 | **社区渗透** | 攻击者长期潜伏于开源社区 → 获得信任 → 植入后门 | XZ Utils（差点感染全球 SSH） |

## 五、密码学攻击面

见 `crypto/05-cryptography-security.md` 第 7~10 节，直接总结如下：

| # | 攻击方向 | 本质原因 | 典型案例 |
|---|---------|---------|---------|
| 1 | **填充预言攻击** | 服务器泄露填充验证结果 → 1 bit oracle → 逐字节恢复明文 | POODLE、Lucky13、Bleichenbacher |
| 2 | **时间侧信道** | 密码操作耗时不同 → 推导密钥/数据 | MAC 逐字节比较、RSA 模幂运算时间 |
| 3 | **弱随机数** | 未初始化熵源/熵不足 → 可预测密钥 | Debian OpenSSL bug (2008) |
| 4 | **nonce 重用** | GCM nonce 重用 → 认证密钥泄露 | AES-GCM 灾难性失败 |
| 5 | **降级攻击** | 兼容性需求 → 遗留协议可用 → 强制使用弱加密 | FREAK、LOGJAM、DROWN |
| 6 | **证书验证缺失** | 不验证 CN/SAN/过期/证书链 → MITM | 自签名证书可接受攻击 |
| 7 | **JWT 算法混淆** | 信任客户端提供的算法名 → none 绕过/none→不对称 | JWT 认证绕过 |

## 六、领域间交叉攻击面

### 6.1 Web + 密码学交叉

```
场景: Web 应用使用 JWT 做认证

  Web 攻击 → 窃取 JWT
  密码学攻击 → JWT 签名绕过
  组合 → 完全控制认证

真实案例:
  - JWT alg: none → 认证绕过（Web + 密码学）
  - JWT RS256 读取公钥 → 用公钥做 HMAC 密钥（算法混淆攻击）
  - JWT kid 注入 → 路径遍历读本地私钥
```

### 6.2 OS + 网络交叉

```
场景: 容器网络

  容器的网络：
    - namespace 隔离了网络栈（OS）
    - veth pair 连接到宿主机（网络）
    - 内核转发（OS | 网络）

  可推导的攻击：
    - 容器内 sock 创建 → sk_buff 分配 → 内核 → 宿主机
    - 如果网络协议解析在内核有漏洞 → 容器逃逸（跨领域）
    - 实际: CVE-2023-26918: TCP 栈漏洞导致容器逃逸
```

### 6.3 开发安全 + 网络交叉

```
场景: HTTP 请求解析

  HTTP 解析器是用 C/C++/Rust 写的
  C 解析器 → 边界检查不当 → 缓冲区溢出 → RCE
  Rust 解析器 → unsafe 块中越界 → 同样问题

  可推导的攻击：
    - 所有网络协议解析器都是潜在的 RCE 向量
    - 解析器越复杂 → 漏洞越多
    - SMB、HTTP/2、DNS 解析 → 经典漏洞来源
```

### 6.4 开发安全 + 密码学交叉

```
场景: 安全随机数生成

  密码学要求: 真随机数
  C 语言: rand() 不是加密安全的
  getrandom()、/dev/urandom 才是

  可推导的攻击：
    - 误用 rand() 生成密钥 → 可预测密钥
    - 忘记初始化熵池 → 所有密钥相同（Debian bug）
    - 容器/VM 恢复快照 → 熵池状态重复
```

## 八、对渗透测试的实战价值

### 这个文档怎么用？

```
面对一个目标时，按攻击面地图逐层思考:

  第1步: 目标打开哪些端口？
    → 参考网络层攻击面 → 选择协议（HTTP/DNS/SSH/SMB）

  第2步: HTTP 服务上有什么？
    → 参考 Web 层 → 找 XSS/SQLi/SSRF/反序列化

  第3步: 认证机制是什么？
    → 参考认证层 → 找 JWT/Session/OAuth 问题

  第4步: 用的是什么语言/框架？
    → 参考代码层 → C 找溢出、Java 找反序列化、JS 找 XSS

  第5步: 用了哪些第三方依赖？
    → 参考供应链层 → 依赖混淆/CVE 查询

  第6步: 加密实现如何？
    → 参考密码学层 → TLS 配置/JWT/随机数检查
```

### 与 CVE 数据库的关系

```
  CVE 告诉你: libcurl 7.83.0 有 CVE-2023-38545 (SOCKS5 堆溢出)
  这个文档告诉你: 所有协议解析器（包括 HTTP、SOCKS、DNS）
    都是用 C 写的，都处理用户输入 → 都可能有堆溢出

  差别: CVE 是"已知的漏洞"
       攻击面推导是"注定的脆弱性"

  实战中两者结合:
    攻击面推导 → 选目标点
    CVE 查询 → 确认已知漏洞
    两者都没有 → 真正的 0-day 探索
```

## 六、新增攻击面（基于实战发现）

以下攻击向量是实战中发现、原 Level 1 文档未覆盖的领域。按重要性排序：

### 6.1 浏览器密码存储 → 攻击面

**信任假设分析：**
| 假设 | 现实 | 攻击方向 |
|:----|:-----|:---------|
| DPAPI 加密足够安全 | 同一用户在同一机器上可直接解密 | 拿到用户权限 → 调用 CryptUnprotectData |
| 加密密钥在 Local State 中安全 | Local State 是明文 JSON 文件 | 只需读取文件即可获取加密后的密钥 |
| 需要用户登录才能解密 | 用户已登录时 DPAPI 凭据在内存中 | 已控权限直接解密 |

**可推导的攻击方向：**
1. **文件系统泄露 → 密码导出** — 通过 Everything HTTP / SMB / RCE 读取 Local State + Login Data → 本地/远程解密所有保存密码
2. **远程凭据收集** — 使用 dploot/SharpChrome 通过 SMB 远程提取浏览器密码
3. **DPAPI 主密钥窃取** — 通过 Mimikatz/SharpDPAPI 提取所有用户的 DPAPI master key → 解密所有 DPAPI 保护数据
4. **密码复用分析** — 获取密码后分析复用模式 → 用同一密码登录其他系统

**关联文档：** `web-security/04-browser-password-security.md`

### 6.2 Everything HTTP（无认证文件系统）→ 攻击面

**信任假设分析：**
| 假设 | 现实 | 攻击方向 |
|:----|:-----|:---------|
| 文件搜索工具只需监听本地 | 默认监听 0.0.0.0 | 局域网任何人都可访问 |
| 搜索工具不需要认证 | 默认无任何访问控制 | 完全开放 |
| 不会暴露敏感文件 | 可浏览所有 NTFS 卷 | 读取 SSH Key/密码库/.env/全部文件 |

**可推导的攻击方向：**
1. **SSH Key 泄露** → `authorized_keys` 读取 → 获得目标公钥 → 找到匹配私钥 → 直接 SSH 登录
2. **浏览器密码文件泄露** → 下载 Local State + Login Data → 离线解密 141 条密码
3. **密码管理器文件泄露** → 下载 .kdbx/.kdb → 离线暴力破解密码
4. **源代码/配置文件泄露** → .env / config.php / docker-compose.yml → 全部凭据
5. **搜索功能利用** → 搜索 `id_rsa` `password` `token` `secret` → 秒级定位敏感文件
6. **全量文件下载** → 整个文件系统作为攻击者资产

**关联文档：** `web-security/05-everything-http-security.md`

### 6.3 Docker 容器安全 → 攻击面

**信任假设分析：**
| 假设 | 现实 | 攻击方向 |
|:----|:-----|:---------|
| 容器内有独立 root → 安全 | 默认无 User Namespace，容器内 root = 宿主机 uid 0 | 容器内 root 可执行特权操作 |
| 容器间网络隔离 | 同一 bridge 网络默认全部互通 | 容器 A → 容器 B 的所有端口 |
| Docker socket 只在宿主机 | 频繁被挂载到容器 | 容器内 docker 命令 → 宿主机全控 |
| 容器逃逸需要0day | 配置错误可逃逸（docker.sock/SYS_ADMIN） | 设计级漏洞而非代码级 |

**可推导的攻击方向：**
1. **docker.sock 挂载 → 容器逃逸** — 挂载 /var/run/docker.sock → docker run --privileged → 宿主机全控
2. **Capabilities 逃逸** — CAP_SYS_ADMIN → nsenter → 宿主机命名空间；CAP_SYS_PTRACE → ptrace 注入
3. **宿主机目录挂载 → 直接逃逸** — `/` 或 `/etc` 挂载 → 直接读写宿主机文件系统
4. **容器内网扁平化** — 同 bridge 网络容器间无防火墙 → Redis/DB 全线暴露
5. **cgroup 逃逸** — release_agent + notify_on_release → 任意代码执行
6. **运行时漏洞** — CVE-2019-5736 / CVE-2024-21626 → 无需配置错误即可逃逸

**关联文档：** `network/03-docker-container-security.md`

### 6.4 密码存储（KeePass/KDBX）→ 攻击面

**信任假设分析：**
| 假设 | 现实 | 攻击方向 |
|:----|:-----|:---------|
| KDBX 文件加密 → 无法读取 | 加密可被离线暴力破解 | 文件泄露 → 离线密码破解 |
| 强密码很安全 | 密码复用使强密码也失效 | 一处泄露的密码可解锁 KDBX |
| KeePass 运行时自动锁定 → 安全 | 剪贴板监控 / 内存转储可提取 | 运行时攻击 |

**可推导的攻击方向：**
1. **KDBX 文件泄露 → 离线破解** — 获取文件 → keepass2john 提取哈希 → hashcat 暴力破解
2. **密码复用 → 已知密码解锁** — 从浏览器/其他文件获得的密码直接尝试解锁 KDBX
3. **内存转储** — procdump KeePassXC.exe → strings 搜索明文密码
4. **剪贴板监控** — 监控 KeePass 粘贴的密码

**关联文档：** `crypto/06-password-storage-security.md`

### 6.5 ZTNA/零信任架构 → 攻击面

**信任假设分析：**
| 假设 | 现实 | 攻击方向 |
|:----|:-----|:---------|
| ZTNA 不暴露端口 → 安全 | 控制平面/身份认证平台仍在运行 | 控制平面 API 可被攻击 |
| 预认证 Key 安全 | Key 有效期可能为年/永久 | Key 泄露 → 任意节点注入 |
| 零信任防止横向移动 | 节点间隧道仍然可达 | 节点内攻击 |

**可推导的攻击方向：**
1. **预认证 Key 泄露 → 节点注入** — tailscale up --authkey → 注册节点到 ZTNA 网络
2. **API Token 泄露 → 完全控制** — 创建/删除节点、修改 ACL 策略、生成永久 Key
3. **IdP 配置错误 → 身份绕过** — Authentik/OIDC SSO 配置错误 → 创建管理员用户
4. **FRP 隧道端口暴露** — FRP 客户端端口扫描 → 发现内部服务
5. **VM 物理级别控制** — Proxmox 宿主机 → 启动 ZTNA VM → 磁盘修改/密码重置

**关联文档：** `network/04-ztna-zero-trust-security.md`

### 6.6 NSG 防火墙 → 攻击面

**信任假设分析：**
| 假设 | 现实 | 攻击方向 |
|:----|:-----|:---------|
| 管理口只能从特定网络访问 | 部署在家庭网络 | 内网中任意机器可访问管理口 |
| 防火墙密码安全 | 密码哈希可被离线篡改 | VM 磁盘挂载 → 修改 sgbase.db 密码哈希 |
| 验证码阻止暴力破解 | 验证码可被绕过 | 验证码生成逻辑缺陷 |

**可推导的攻击方向：**
1. **默认凭据尝试** — admin/admin、admin/nsg_admin、root/password
2. **验证码绕过** — 可预测的验证码生成、Session 固定、空验证码
3. **VM 磁盘离线篡改** — Proxmox 挂载 VM 磁盘 → 修改 SQLite 密码哈希 → 用已知密码登录
4. **SSH 暴力破解** — 使用已知密码字典尝试 SSH 登录
5. **SNMP 信息泄露** — public/private community → 网络拓扑暴露
6. **防火墙策略绕过** — 管理口 ALLOW ANY → 通过防火墙自身访问受限网络

**关联文档：** `network/05-nsg-firewall-security.md`

### 最重要的一个结论

> **所有漏洞都源于一个基本矛盾：系统设计时的「信任假设」和实际运行时的「现实情况」之间的差异。**
>
> 系统假设 X 是安全的 → 实际上 X 不安全 → 漏洞。
>
> - 假设: 网络包来源可信 → 实际: 任何人可发 → 伪造
> - 假设: 用户输入是文本 → 实际: 用户输入 <script> → XSS
> - 假设: 包名唯一 → 实际: 谁都能注册 → 依赖混淆
> - 假设: 序列号随机 → 实际: 16-bit → 劫持
> - 假设: 签名者可信 → 实际: 构建系统被攻破 → SolarWinds
>
> 渗透测试者最重要的技能不是知道 CVE 怎么用，
> 而是**理解每个假设，并问「如果这个假设不成立会怎样」。**

## 【LLM 推理段 — 综合侦查推理流程】

### 通用侦查步骤（按目标类型）

#### Web 服务（端口 80/443/8080）
1. 确定技术栈：whatweb/nmap 识别 CMS/框架/版本
2. 扫描目录：gobuster/dirsearch 找隐藏路径
3. 检查安全头：CSP/HSTS/X-Frame-Options/CORS
4. 测试输入点：所有参数/表单/API 端点
5. 检查认证：登录/注册/密码重置/OAuth
6. 检查文件上传：图片/文档/头像

#### 内网主机（任意端口开放）
1. 识别服务版本：nmap -sV
2. 检查默认凭证：admin/admin、root/root
3. 检查已知 CVE：版本号匹配
4. 检查开放共享：SMB/NFS/FTP 匿名访问
5. 检查 RDP/SSH 配置：是否暴露、弱口令

#### 云服务（S3/OSS/Cloud API）
1. 检查存储桶权限：公共读写、枚举
2. 检查云元数据：SSRF 到 169.254.169.254
3. 检查 API Key 泄露：代码/配置文件/Git

### 攻击面优先级排序
| 优先级 | 攻击面 | 原因 |
|--------|--------|------|
| 🔴 极高 | 远程代码执行/命令注入 | 直接拿 shell |
| 🔴 极高 | SQL 注入 | 数据泄露 + 可能 RCE |
| 🟡 高 | 认证绕过/越权 | 账号接管 |
| 🟡 高 | 文件上传/包含 | 可能 RCE |
| 🟢 中 | XSS/CSRF | 需组合利用 |
| 🟢 中 | 信息泄露 | 扩大攻击面 |
| ⚪ 低 | 配置错误 | 影响不大但有组合价值 |

### 确认漏洞后的三步决策
1. **能否直接利用** -> 能否拿到 shell/数据/权限？
2. **能否扩展攻击面** -> 这个漏洞能让我接触到什么之前看不到的？
3. **能否组合利用** -> 这个漏洞 + 其他漏洞 = 更大的危害？
