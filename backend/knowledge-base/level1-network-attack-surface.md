# 网络协议攻击面推导 — 从协议设计到攻击向量

> 关联 Level 0: `protocols/01~06`（TCP/IP, HTTP, TLS, DNS, SSH, SMB）
> 关联 Level 2: Case 1 (EternalBlue), Case 10 (Heartbleed), Case 12 (PetitPotam)
>
> 目标：枚举每个协议的信任假设，标注假设被突破后的攻击向量。

---

## 一、TCP/IP 协议栈的信任假设

### 信任假设 A：IP 地址是身份标识

- **假设内容**：源 IP 地址标识了发送者的身份。
- **机制来源**：IP 头部包含源地址和目的地址，路由只按目的地址转发。
- **如果被突破**：攻击者可伪造源 IP → 冒充其他主机。

| 突破方式 | 推导的攻击 | 实际案例 |
|---------|-----------|---------|
| IP 头部源地址可任意填写（路由不验证） | IP 欺骗 → 非授权访问 | 传统 rsh/rlogin 信任 IP |
| 结合 TCP 序列号预测 | IP 欺骗 + TCP 会话劫持 | Mitnick 攻击 (1994) |
| 结合序列号嗅探 | TCP 劫持 → 注入恶意数据 | 中间人攻击 |

**推理模式：** IP 地址是"位置"标识，不是"身份"标识。任何依赖 IP 做认证的机制（防火墙白名单、基于 IP 的信任）都可以被伪造。攻击者只需在网络路径上或能伪造源 IP 的位置即可。

### 信任假设 B：TCP 序列号不可预测

- **假设内容**：TCP 序列号随机化（RFC 6528），攻击者无法预测。
- **机制来源**：初始序列号 (ISN) 由内核随机生成。
- **如果被突破**：攻击者可在不在路径上的情况下伪造 TCP 连接。

| 突破方式 | 推导的攻击 | 实际案例 |
|---------|-----------|---------|
| ISN 生成算法弱（旧内核） | TCP 会话伪造 → 无认证 RCE | Mitnick 攻击中的序列号预测 |
| ISN 时序侧信道 | TCP 注入 → 重置连接 / 污染数据 | Sockstress, TCP RST 攻击 |
| 序列号回绕 | 重放旧数据段 → 混淆数据 | TCP 时间戳防御机制 |

**推理模式：** TCP 序列号是"连接完整性"的基石。如果序列号可预测或可嗅探 → 攻击者可以在数据流中注入伪造数据包。

### 信任假设 C：TCP 状态机正确跟踪连接

- **假设内容**：两端 TCP 状态机同步，SYN→SYN-ACK→ACK 三次握手建立连接。
- **机制来源**：有限状态机，转换由标志位触发。
- **如果被突破**：状态机不一致 → 资源耗尽 / 连接劫持。

| 突破方式 | 推导的攻击 | 实际案例 |
|---------|-----------|---------|
| 发大量 SYN 但不完成握手 | 连接表填满 → 拒绝服务 | SYN Flood |
| 发 RST 伪装 | 断开任意连接 → DoS | TCP RST 攻击 |
| 发 SYN 但 IP 不存在 | 反射放大 → 带宽耗尽 | SYN Flood 变种 |

**推理模式：** TCP 状态机假设对端会按规则行事。但攻击者可以发送任何标志位组合。状态机设计必须考虑"不完整的状态"——即收到不符合预期的包时怎么办。

## 二、HTTP 协议的信任假设

### 信任假设 D：请求和响应是一一对应的

- **假设内容**：HTTP 请求与响应在 HTTP 层是一一对应关系。
- **机制来源**：HTTP/1.1 默认 Keep-Alive，同一连接上顺序发送请求。
- **如果被突破**：请求和响应边界混淆 → 请求走私。

| 突破方式 | 推导的攻击 | 实际案例 |
|---------|-----------|---------|
| 前端后端对 Content-Length 和 Transfer-Encoding 的优先级理解不同 | HTTP 请求走私 → 前端认为一个请求，后端认为两个 | 各类 HTTP 走私变种 |
| 请求体长度计算不一致 | 前端放行 → 后端处理额外请求 | 缓存投毒、用户劫持 |

**推理模式：** 代理架构下，前端解析和后端解析的不一致 = 攻击面。因为信任"前端怎么解析，后端就怎么解析"是不成立的——不同 HTTP 实现有不同的 RFC 实现细节。

### 信任假设 E：HTTP 头部包含足够的认证信息

- **假设内容**：Cookie、Authorization 头足以标识用户身份。
- **机制来源**：HTTP 是无状态的，状态由头部携带。
- **如果被突破**：头部信息泄露或被伪造。

| 突破方式 | 推导的攻击 | 实际案例 |
|---------|-----------|---------|
| Cookie 未标记 Secure/HttpOnly | XSS → 窃取 Cookie → 会话劫持 | 会话固定、会话劫持 |
| CORS 配置允许任意 Origin | 跨站读取响应 → 数据窃取 | CORS 配置错误 |
| Cookie Domain 范围过大 | 子域名 XSS → 窃取父域 Cookie | 子域名接管 |

**推理模式：** 无状态协议把安全负担转移给了"认证令牌"本身。令牌的安全性 = 令牌的生成机制 + 传输渠道 + 存储方式的最小值。

## 三、TLS 的信任假设

### 信任假设 F：证书链验证可靠

- **假设内容**：CA 只给合法实体签发证书，证书验证会检查整个链。
- **机制来源**：X.509 证书链 → 根 CA → 中间 CA → 服务器证书。
- **如果被突破**：攻击者可获得对任意域名的有效证书。

| 突破方式 | 推导的攻击 | 实际案例 |
|---------|-----------|---------|
| CA 被攻破 → 签发伪造证书 | 中间人攻击 → 解密所有流量 | DigiNotar (2011, 荷兰 CA) |
| CA 签发时未验证域名所有权 | 获得他人域名的证书 | Let's Encrypt 早期验证问题 |
| 客户端未验证证书链 | 自签名证书也被接受 | 私有 CA 配置错误 |
| 证书固定 (pinning) 被绕过 | 信任新的伪造证书 | Superfish (Lenovo) |

**推理模式：** TLS 的安全依赖 CA 系统的安全。CA 系统是中心化的信任根——一个 CA 被攻破 = 整个 PKI 系统对该域名的信任被破坏。

### 信任假设 G：TLS 实现正确实现了协议

- **假设内容**：OpenSSL/BoringSSL 等实现严格遵循 RFC。
- **机制来源**：实现细节（长度检查、状态转换）。
- **如果被突破**：实现错误 → 协议安全失效。

| 突破方式 | 推导的攻击 | 实际案例 |
|---------|-----------|---------|
| 未验证心跳请求长度 | 越界读 → 内存泄露 | Heartbleed (CVE-2014-0160) |
| 填充验证顺序错误 | 填充预言 → 渐进解密 | POODLE (CVE-2014-3566) |
| 错误消息处理不当 | 降级攻击 → 使用弱加密套件 | DROWN (CVE-2016-0800) |

**推理模式：** 密码学协议的正确性取决于实现的正确性。TLS 实现是"复杂状态机 + 密码学"的组合——状态机错误比密码学错误更常见。

## 四、DNS 的信任假设

### 信任假设 H：DNS 响应来自真正的服务器

- **假设内容**：DNS 查询得到的结果来自权威服务器。
- **机制来源**：UDP 无连接；缓存 DNS 服务器转发查询。
- **如果被突破**：DNS 响应可被伪造 → 域名解析到攻击者 IP。

| 突破方式 | 推导的攻击 | 实际案例 |
|---------|-----------|---------|
| UDP 源地址可伪造 | DNS 响应伪造 → 缓存投毒 | Kaminsky 攻击 (2008) |
| TXID 16位可暴力 | TXID 碰撞 → 伪造响应 | 经典 DNS 投毒 |
| DNSSEC 部署率低 | 响应可篡改 → 中间人 | 公共 WiFi DNS 劫持 |

**推理模式：** DNS 的设计假设"网络是可信的"——这在不安全的互联网上不成立。16位 TXID 的随机空间太小，不足以防伪造。DNSSEC 虽然解决了验证问题但部署率不足。

### 信任假设 I：DNS 服务器是诚实的

- **假设内容**：递归 DNS 服务器不会篡改查询结果。
- **机制来源**：客户端信任配置的 DNS 服务器。
- **如果被突破**：DNS 服务器可返回任意结果。

| 突破方式 | 推导的攻击 | 实际案例 |
|---------|-----------|---------|
| ISP DNS 服务器被篡改 | 广告注入、钓鱼重定向 | 部分 ISP 的 NXDOMAIN 劫持 |
| 公共 DNS 被监控 | 查询日志泄露 → 访问记录暴露 | DNS 隐私问题 |
| 恶意 DNS 服务器 | 域名劫持 → 所有流量可控 | 路由器 DNS 篡改 |

**推理模式：** 你信任的 DNS 服务器控制了你看到的整个互联网。任何能修改 DNS 配置的攻击（路由器漏洞、DHCP 劫持）都相当于控制了所有后续的流量。

## 五、SSH 的信任假设

### 信任假设 J：主机密钥首次使用时验证

- **假设内容**：用户首次连接 SSH 时，会验证主机密钥指纹。
- **机制来源**：SSH 协议设计：首次信任 (TOFU — Trust On First Use)。
- **如果被突破**：中间人攻击在首次连接时无法检测。

| 突破方式 | 推导的攻击 | 实际案例 |
|---------|-----------|---------|
| 首次连接时不验证指纹 | 中间人 → 密码泄露 | 典型 TOFU 攻击 |
| 主机密钥泄露 | 冒充服务器 → 解密历史流量 | 密钥轮换不足 |
| SSHFP DNS 记录未使用 | 无法自动验证主机密钥 | 部署不足 |

**推理模式：** TOFU 的设计假设"首次连接的环境是安全的"——这不适用于不安全的网络（公共 WiFi）。SSH 的信任模型缺少带外验证机制。

### 信任假设 K：Agent Forwarding 只在信任的主机上使用

- **假设内容**：用户只在信任的远程主机上启用 Agent Forwarding。
- **机制来源**：SSH Agent 使用 Unix 域套接字转发认证私钥的签名请求。
- **如果被突破**：恶意管理员（或攻破了跳板机）可通过 Agent 使用用户的私钥。

| 突破方式 | 推导的攻击 | 实际案例 |
|---------|-----------|---------|
| 跳板机被攻破 → Agent 套接字可访问 | 使用用户私钥签名 → 横向移动到其他主机 | 经典跳板机攻击 |
| 用户忘记关闭 Agent Forwarding | 攻击链中的跳跃点 | 渗透测试中的常用技术 |

**推理模式：** Agent Forwarding 是把"能使用私钥的能力"传递给了远程主机。任何能访问 Unix 套接字的进程都可以代表用户签名。这相当于把家门钥匙交给了跳板机的管理员。

## 六、SMB 的信任假设

### 信任假设 L：SMB 协议实现正确验证消息格式

- **假设内容**：SMB 服务器正确处理消息格式、大小和边界。
- **机制来源**：SMB 是复杂协议（数百种命令），各版本差异大。
- **如果被突破**：内存破坏 → 远程代码执行。

| 突破方式 | 推导的攻击 | 实际案例 |
|---------|-----------|---------|
| 整数截断（DWORD→WORD） | 缓冲区分配不足 → 堆溢出 | EternalBlue (CVE-2017-0144) |
| 未验证 SMBv1 兼容性 | 利用遗留代码中的漏洞 | EternalBlue 触发路径 |
| 复合请求处理错误 | 状态机混乱 → 非预期行为 | SMBGhost (CVE-2020-0796) |

**推理模式：** 遗留协议兼容性是攻击面放大器。SMBv1（1980年代设计）在不安全网络上设计，保留在 Windows 中数十年 → 成为最严重的远程漏洞入口。

### 信任假设 M：NTLM 认证是安全的

- **假设内容**：NTLM 挑战-响应机制能安全验证身份。
- **机制来源**：NTLM 使用密码哈希 + 挑战值做响应。
- **如果被突破**：凭据可被中继、重放或离线破解。

| 突破方式 | 推导的攻击 | 实际案例 |
|---------|-----------|---------|
| NTLM 无通道绑定 | 中继到其他服务 → 冒充 | SMB Relay, PetitPotam |
| NTLMv1 使用弱 DES | 离线暴力破解 → 明文密码 | 经典 NTLM 破解 |
| Net-NTLMv2 可被中继 | 跨协议中继 → 接管域控 | PetitPotam → AD CS |

**推理模式：** NTLM 设计时假设"网络是可信的，攻击者不在路径上"。但在现代网络中，攻击者经常在网络路径上或能引诱目标主动连接。

## 七、跨协议攻击面

### 协议转换时的信任断层

| 转换 | 信任假设 | 攻击面 |
|------|---------|--------|
| HTTP → TLS (HTTPS) | 浏览器信任证书 | 但证书验证逻辑可能在 HTTP 层被绕过（如混合内容） |
| DNS → HTTP | 浏览器信任 DNS 解析结果 | DNS 投毒 → 钓鱼 |
| HTTP → SMB | 浏览器处理 file:// 协议 | 本地文件泄露 |
| SSH → Agent → SMB | Agent Forwarding 链 | 跳板机到手后横向移动到 Windows |

**推理模式：** 协议栈的每一层都有信任假设。层与层之间的接口 = 信任假设的传递链。如果一个低层假设被突破，所有上层都受影响。

## 八、实战指导

```
从网络协议角度评估一个目标：

1. 暴露了哪些端口？
   → 每个端口 = 一个协议的实现 = 一组信任假设
   
2. 每个协议使用了什么版本？
   → 旧版本 = 旧时代的信任假设（如 SMBv1、TLS 1.0）
   
3. 认证机制是什么？
   → 基于 IP 的信任？基于密码？基于证书？
   → 每个认证机制有一组信任假设

4. 协议实现是什么？
   → OpenSSL 3.x vs 1.0.2? Samba vs Windows SMB?
   → 实现 = 信任假设的兑现方式

5. 协议之间如何交互？
   → DNS → HTTP → API 调用
   → 一个协议的输出是另一个协议的输入
   → 信任假设在这里传递
```
